当 AI 接管 A/B 消融实验：一次提示词优化引发的思考

我写了个英语应用，用 AI 辅助我弟练习英语：AI 造句，人翻译。



但我发现，语音识别会误听同音字，导致接下来 LLM 识别不准（比如误判男女Ta之类）。于是我让 AI 根据判题日志，找到此类日志、设计测试用例，自己测试让 LLM 提示词覆盖这些边界。然后再用奥卡姆剃刀，设计迭代消融实验尽可能简化提示词。它实验了 39 个版本，调用 1180 次 DeepSeek API 测试评估，最终在通过所有用例的情况下，将提示词短了 60%，让后续每次调用 Token 成本降低 44%。

往大了想，不只是提示词优化——产品里那些按钮颜色、文案措辞、图标位置的 A/B 实验，AI 同样可以自主闭环，把迭代周期从周级压缩到分钟级。



人类负责定义优化目标和初始策略，AI 负责从初始策略出发，想办法向着目标自我迭代到局部最优。



起因：一个语音判题的 Bug

整个链路是这样的：

flowchart LR
    A[用户说中文] --> B[ASR（语音识别）转写为文字]
    B --> C[LLM 判题器评判对错]
    C --> D{是否正确?}
    D -- 正确 --> E[给出正确反馈]
    D -- 错误 --> F[给出错误反馈]

触发 bug 的那条记录：

英文句子：Fame brings responsibility.

用户说的是「名声带来责任」—— 答对了

ASR 转写结果：「民生带来责任感」—— ASR 把「名声」听成了「民生」

系统判定：❌ 错误

AI 反馈：「"民生"和"名声"发音很像，但意思差远了哦。」

用户答对了，系统判错了。 「名声（míngshēng）」和「民生（mínshēng）」发音极近，是 ASR 的识别误差。但判题器的提示词里没有处理这类情况的容错逻辑，所以照样判错。

类似的还有：「迈克尔」→「麦克尔」、「她的」→「他的」（普通话 tā 同音）、「影响」→「印象」。

这些全是 ASR 同音误转，不是用户翻译错误。 问题出在提示词——它只提到了代词（他/她）的同音容错，没有覆盖实词和音译人名。

第一反应：打补丁还是做实验？

最直觉的修法是加规则：「遇到民生/名声，容错处理」。但这条路走下去是这样的：

- 民生↔名声：容错
- 麦克尔↔迈克尔：容错
- 他↔她：容错
- ……（每发现一个 case，加一条）


这是典型的「打补丁」——提示词最终变成一堆 if-else，越来越长、越来越脆。

这些场景的本质是同一类问题：ASR 同音误转。 不应该枚举词对，而应该让模型理解一条通用原则。但怎么验证有没有效？靠感觉写完就上线？我提出了方法：用 benchmark 做实验，不要靠感觉。从这一步开始，执行层面的工作全部交给了 AI。

AI 自主完成了：设计 26 个测试用例、编写自动化评估脚本、设计 8 轮消融实验矩阵、执行 1,180 次 API 调用、分析数据、修正标注错误、最终组合出最小可行版本并连跑 3 次确认稳定性。我只在几个关键节点做方向判断——要不要打补丁、要不要继续压缩、最终选哪个版本上线。

实验执行细节（8 轮消融实验，39 个版本）：

测试集设计
──────────
AI 设计了 26 个测试用例，覆盖 6 类场景：
- 明确判对（5例）：语义正确的各类表达（同义、口语化）
- 明确判错（5例）：否定漏译、语义相反、缺主干
- ASR 单字同音（4例）：「她→他」类代词误转
- ASR 整词同音（5例）：「名声→民生」「影响→印象」类实词误转
- 音译人名变体（4例）：「迈克尔↔麦克尔」类外来词
- 模糊边界（3例）：近义词替换、口语化表达

自动化评估脚本支持：批量发送用例、对比输出与预期、反馈文字质检、多版本并发测试。

第 0 轮：架构方向（3 个版本）
──────────────────────────
AI 把提示词重构成「两步走」框架——先做 ASR 纠错，再判语义。
结果：准确率从 96.2% 暴跌至 73.1%。
原因：两步走框架让模型以为要「向用户汇报纠错过程」，把「他/她」差异当作错误输出。
结论：看起来更清晰的提示词，效果更差。

第 1 轮：规则 vs 示例（3 个版本）
──────────────────────────────
- 只有规则，无示例 → 92.3% + hint bug（无示例时模型会「说废话」）
- 只有示例（10个），无规则 → 92.3%（示例多≠效果好）
- 规则+示例（4个） → 100%（两者缺一不可）
结论：规则建立通用原则，示例锚定具体行为。两者是协作关系，不能相互替代。

第 2-3 轮：逐部分消融（11 个版本）
────────────────────────────────
对最优版本的每个组成部分逐一删除，测量贡献：
- 规则 6 内联示例（2行）：删后 −3.9%，不可删，是代词容错的锚点
- 规则 6 词对列举：删后不变，可删，内联示例已足够
- 反馈规则（7条→3条）：删后不变，可压缩
- 规则 7-8（冗余逻辑）：删后不变，可删
- 全部示例：删后 −3.9% + bug，不可删
- 字段名重命名（contextualTranslation→tr）：不变，每次请求省~15 tokens
- XML 格式→行内格式，4例→2例：不变，格式不影响理解

第 A-G 轮：极限压缩（22 个版本）
──────────────────────────────
逐条规则测试显示「每条单独删都不影响准确率」。全部删掉后：96.2% → 80.8%。
结论：单条可删，整体不能删。单变量测试不能预测多变量结果。

还测试了 header 压缩（3行→1行）、反馈合并（3条→1行）、示例减量（2→1→0）等。
最终组合出最小可行版本，连跑 3 次确认稳定性。

人类介入点
─────────
- 开始：发现问题，判断「这是一类问题」，提出做 benchmark
- 第 0 轮后：看到数据，指出「不应该为每个 case 打补丁」
- 第 2 轮后：要求「去做消融实验，不要靠猜」
- 第 3 轮后：要求「继续压缩到极限」
- API 余额用完：充值，让 AI 继续
- 实验全部完成：确认最终版本，批准部署上线


提示词对比：优化前 vs 优化后

优化前（V1，1678 字符）：

你是"语音翻译判题器"。
数据链路：用户中文语音 -> ASR 转写文本 -> 你判题。你拿到的是转写结果，不是手打原文。
任务：判断学生作答是否表达了目标英文句子的核心语义。

判题原则：
1. 先做"核心槽位覆盖检查"：动作、对象、否定、关键关系（如归属/方向/比较）必须齐全。
2. 只说关键词、缺主干、缺关键对象或关键关系时，直接判错（correct=false）。
3. 允许省略不影响句意的非核心修饰（语气词、轻微口语补充）。
4. 仅在"核心槽位完整"前提下，再判断语义是否一致。
5. ASR 噪声默认容错：同音/近音字、错别字、意外标点、口语词、重复词、轻微语序差异。
6. 先做"同音容错推理"：遇到可疑词，先尝试同音/近音替换并代回整句语义，再决定对错。
   - 常见代词同音示例：他/她/它/TA、他的/她的/它的。
   - 示例：目标句意是"这个书包是她的"，ASR 转写成"这个书包是他的"时，
     默认先按同音误转处理，不直接判错。
7. 若容错后能形成"核心槽位完整且一致"的中文表达，判对（correct=true）。
8. 只有在容错后仍然核心语义错误或覆盖不完整时才判错。

反馈：
1. correct=true：teacherHint 设为 ""（空字符串）。
2. correct=false：只指出 1 个最关键错误，并给可执行改法。
3. 禁止把单个同音字差异当成错误点；要聚焦句意层面的关键差异。
4. contextualTranslation 用当前语境下自然中文整句，不用"TA"字母写法。
5. teacherHint 要有人味，允许情绪和轻微吐槽，但不能每次都一个语气：
   - 偶发错误：偏鼓励、简短提醒。
   - 同类错误反复出现：更直接，带一点调侃。
   - 连续多次犯同类错：可以半开玩笑地"点醒"。
6. 禁止人身攻击、羞辱、阴阳怪气过头；只调侃错误，不调侃人。
7. teacherHint 控制在 10~26 字，口语化，别写成课堂讲稿。

  <example>
    <input>英文句子: This backpack is hers. 学生作答: 这个书包是他的。</input>
    <output>{"correct":true,"contextualTranslation":"这个书包是她的。","teacherHint":""}</output>
  </example>

四个反直觉发现

1. 更长的提示词不一定更好

10 个示例（2111字符）的准确率比 2 个示例（911字符）更低。堆内容不是解法，精准才是。

2.「看起来更清晰」的结构可能更差

两步走框架逻辑上完美，实际准确率暴跌 23%。LLM 理解语言的方式和人不同——改了结构就改了模型的推理路径。

3. 单变量测试不能预测多变量结果

规则 1-5 每条单独删都不影响准确率。全部删掉，准确率从 96.2% 跌到 80.8%。这就是为什么需要做组合消融，而不是逐条测完就下结论。

4. 规则和示例是协作关系

规则提供泛化能力（「遇到同音词要容错」），示例锚定具体行为（「他/她 → 判对」）。只有规则或只有示例都是 92.3%，组合在一起才能到 96.2%+。

这套方法可以用在哪里？

回到核心观点：你指出问题，AI 搜索最优解。

你要做的只有一件事——告诉 AI「这里有个问题，去优化」。测试用例怎么设计、评分标准怎么定、消融矩阵怎么排、数据怎么分析——全部由 AI 自主完成。

这本质上是一个搜索问题：在「所有可能的提示词变体」这个巨大的空间里，搜索出一个成本最低、准确率最高的解。消融实验就是搜索策略——每次缩小一个维度，观察效果是否退化，逐步收敛到最优点。

传统做法是凭经验改提示词、凭感觉判断效果。现在可以把这个过程变成：

① 你定义评分标准（测试集 + 成功指标）
② AI 自动搜索最优解（消融实验 + 数据驱动决策）
③ 你确认上线


适用条件：

有明确的成功指标（准确率、点击率、转化率、满意度评分、甚至另一个 LLM 的打分）

测试可以自动化执行

满足这两点，任何 LLM 驱动的功能——客服问答、内容生成、代码辅助、推荐文案——都可以用这套方式做提示词优化。

这套方法不只适用于提示词优化。往大了想，互联网产品里大量的日常优化工作——按钮用红色还是绿色、标题是否加粗、图标放左边还是右边、文案用「立即购买」还是「马上抢」——本质上都是同一类问题：在一个策略空间里搜索最优解。

过去这些事全靠人来做：产品经理提方案、开发上实验、等一周流量、分析数据、再决定下一步。一个按钮颜色的 A/B 实验，从提出到结论可能要两周。



现在 AI 可以把这整个循环闭环掉。

人类产品经理负责定义优化目标和初始策略，AI 负责从初始策略出发，朝目标方向高速迭代收敛到局部最优。

这就像在优化平面上：人类定义损失函数（「什么叫好」——准确率、成本、转化率）和初始点（第一版提示词或方案），然后由 AI 自主做梯度下降。

传统 A/B 实验是人来驱动的：人设计实验方案、人等流量跑够、人分析数据、人决定下一步改什么。一轮实验周期通常以周为单位。



第一步变化：AI 接管实验循环，但周期不变。 同样是以周为单位跑实验，但 AI 可以全自动地完成过去人做的所有事——自己提出假设、自己验证、自己分析数据、自己迭代策略。而且 AI 可以在每个实验周期内同时开启 N 个策略分支并行测试，不像人只能顾得过来两三个。

第二步变化：把实验周期从「周级」压缩到「分钟级」。 传统实验需要跨周对比流量，是为了保证统计显著性——本周 vs 上周。但仔细想想：这一分钟的流量和上一分钟的流量，真的有本质区别吗？大多数场景下，分钟级别的流量分布差异极小。如果 AI 能以分钟为粒度开实验、回收数据、判断效果，那迭代速度就从「一周一轮」变成「一分钟一轮」。

本文的案例还只是用构造的测试集做离线实验。但如果把这套能力接入线上环境——AI 自动开实验、自动回收用户数据、自动判断效果并迭代策略——那传统 A/B 实验中那些「设计实验方案、等数据、分析报告、决定下一步」的人力成本，都可以被压缩到接近零。



过去这些繁琐的实验工作之所以需要人来做，是因为没有一个角色能同时「写代码 + 设计实验 + 分析数据 + 迭代方案」。现在 AI 可以。

给不同读者的建议

一线技术开发： 消融实验的脚本逻辑不复杂，核心就是「每次只改一个变量」。输出字段重命名（长字段名→短字段名）这种细节，在 token 定价模式下是真实的成本节省。脚本可以直接复用。

技术团队主管： 优化后每千次请求节省 ¥1.30。如果日均调用 10 万次，年化节省约 ¥47,000。更重要的是：这套能力建立起来之后，每个 LLM 功能都可以走同样的流程。

产品部门负责人： 这不是一次性的工程优化，是一套可复用的方法论。它证明了一件事——当你能定义「什么叫好」并自动化测试时，提示词优化就不再是黑箱里的玄学，而是可以交给 AI 自主执行的工程流程。
