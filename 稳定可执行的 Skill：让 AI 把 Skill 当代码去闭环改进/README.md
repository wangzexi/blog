# 稳定可执行的 Skill：让 AI 把 Skill 当代码去闭环改进

| 单向：写完即止 | 闭环：可迭代验证 |
| --- | --- |
| 1. 人指挥 AI 做一遍<br><br>2. 再让 AI 把过程总结成 Skill | 1. 人指挥 AI 做一遍<br><br>2. 再让 AI 把过程总结成 Skill<br><br>3. AI 委派`独立上下文的 AI`看着这个 Skill，从任务原点重新再做一遍<br><br>4. AI 对比`独立 AI 产出`和`人产出`的差异，再修订 Skill |

目前很多编写 Skill 的是单向的，以至于，你并不知道这个 Skill 到底能不能真正驱动 AI 产出结果。

如果能让它能闭环，多修订几轮，Skill 才能从“看起来合理”变成“实际好用”。


![image](assets/234149479268.svg)


## 具体案例

最近我们发现：AI 写出的代码能跑，但啰嗦，大量散弹枪防御、职责混杂、声明和使用点离得远、Diff 很难 Review…


于是基于我带着 AI 做一系列简化：

> 人工具体 Commit：：
> - 把 `-999` 这类埋点约定下沉到埋点组装层
> 
> - 让 helper 只返回真实结果
> 
> - 把 early return 收直
> 
> - 让 `rootTag` 取到后立刻判断
> 
> - 去掉没有依据的多余防御
> 
> - 去掉无收益注释
> 
> - 把变量和判断尽量贴近使用点


然后再让 AI 总结成 Skill。

最关键是，我们让 AI 派出独立上下文的子 AI，==只给它任务原点和 Skill，让它重新做一遍==。


这样，在子 AI 完成后，把它的结果和人工结果一对比，就能看出：

- Skill 哪些地方已经够用

- 哪些边界还没写清


这次补进去的内容就包括：

- 先收紧文件范围，再做局部收尾

- 删防御前先做代码检索，确认真实来源或写入点

- 只有 helper 职责边界变化真的影响直接调用方可读性时，才允许一起改调用方

- 默认不新增正式测试文件

- `rn-live` 里必须跑 `bun run tscheck`



这样迭代几轮之后，就逐渐变成真正可执行 readability-cleanup。


## 核心思路

Skill 不是文档，而是驱动 Agent 这个执行器的代码。

所以 Skill 不能写了就完了，还要验证可执行，且最简。


1. 先人工指挥 AI 把真实任务做到自己满意

2. 让 AI 总结成初版 Skill，把目标、边界、默认动作、验证方式收成短规则

3. 让 AI 派独立子 Agent 执行，只给 Skill 和和任务原点

4. 再让 AI 对比子 Agent 的结果和人类的结果，发现偏移点

5. 利用执行疏漏修正 Skill
- 边界不清，就补边界

- 原则太虚，就补动作

- 必须依赖事实判断，就写清“先检索再决定”

- 能不新增规则就不新增，优先压回最简表达


6. 多来几轮，直到
- 子 Agent 按照 Skill 从同样的原点，能做到接近人工结果

- Skill 没有越迭代越厚，留下来的规则仍然简洁

- 再继续改，收益已经明显变小
