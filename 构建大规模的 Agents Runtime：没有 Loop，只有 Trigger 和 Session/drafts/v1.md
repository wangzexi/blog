# 生产级 Agents 运行时：Agent = Session + Harness

**Agent = Session + Harness(Model, Tool?, SubAgent?, ...)。**

Session 是持久状态。Harness 是参数化的调度器——Model 是唯一必需的执行器，Tool 和 SubAgent 是可注入的能力。

---

## 为什么容器不能是 Agent

很多 Agent 框架的默认假设：

```
一个 Agent = 一个容器
```

Claude Loop、Browser、Terminal、Files——全部塞进同一个容器。

这很自然。大多数早期框架都这么干。

问题是：**你把生命周期完全不同的两种东西绑死了。**

一种是 Claude 的思考——读消息、推理、决定调用工具。几十 MB 内存，挂了重启就行。

另一种是工具执行环境——Chrome、Node、Python、文件系统。几 GB，随时可能崩。

在操作系统里，没有人会把 CPU 和磁盘焊死在同一个芯片上。CPU 的寿命是年，磁盘的寿命是小时级读写。它们被设计成独立的组件，通过标准接口通信。

Agent 也一样。思考循环和执行环境的故障模式、资源需求、生命周期完全不同。不该放在一起。

---

## 解的理应是脑和手的生命周期

Anthropic 在 2026 年 4 月的 Engineering Blog《Scaling Managed Agents: Decoupling the brain from the hands》里把架构改成了：

```
          Harness
      Claude Loop
            │
            │ tool_call
            ▼
      executeTool()
            │
            ▼
         Sandbox
       (远程容器)
```

Sandbox 从 Agent 本体变成了一个**远程工具**——就像磁盘从 CPU 内部移到了总线上。

容器挂了？那是一次工具调用失败。Claude 创建新 Sandbox，重试。

```
Container crash ≠ Agent crash
Sandbox failure ≠ System failure
```

操作系统里，驱动崩溃不会让内核一起死。Agent 也一样。

## Harness 是内核调度器，不是画家

一旦你接受 Agent Runtime = 操作系统，Harness 的角色就清晰了。

Harness 不产生智能。它不做计算。它的工作是：

```
1. 等中断（Trigger）
2. 读内存（Session）
3. 分配 CPU（调 Model）
4. 路由系统调用（Tool Call）
5. 写回内存（Session）
6. 继续 idle
```

这跟操作系统的调度循环没有区别。

**Harness 就是一个内核调度器。**

Model 是 CPU——执行计算，但不决定自己何时运行。Tool 是设备驱动——执行 I/O，但不参与调度决策。Sub-Agent 是用户进程——运行在 Runtime 之上，被 Harness 统一管理。

## 恢复策略为什么是时间的函数

如果 Agent 的操作系统本质成立，恢复策略自然就是时间函数。

CPU 不会用同一种方式保存所有中断状态：

| 中断类型 | 恢复方式 | 对应 Agent 场景 |
|---------|---------|---------------|
| 寄存器切换 | 立即恢复 | Replay（几秒内的中断） |
| 写内存 | 加载进程状态 | Summary（几分钟到几小时） |
| 写硬盘 | 分页加载 | Checkpoint Recovery（几天以上） |

Agent 也一样：

```
0 ~ 5 分钟         → Replay（热，KV Cache 还在）
5 分钟 ~ 1 天       → Summary（温，Harness 帮压上下文）
1 天 ~ 7 天         → Checkpoint Recovery（冷，Agent 自己恢复）
7 天以上            → Cold Start（冻，只给 Workspace + Checkpoint）
```

这不是拍脑袋。这是 CPU 中断恢复策略在 Agent 世界的直接映射。

## Agent 自己定义下一次怎么醒来

操作系统允许进程注册信号处理函数。Agent 也应该允许自己定义恢复策略。

Agent 在挂起之前，告诉 Harness 它想怎么被唤醒：

```json
{
  "trigger": "ToolFinished",
  "resume_policy": {
    "strategy": "replay",
    "include_recent_events": 20,
    "include_files": ["task.md"]
  }
}
```

定时任务也一样：

```json
{
  "trigger": "TimerExpired",
  "at": "2026-06-20T09:00:00",
  "resume_policy": {
    "strategy": "checkpoint",
    "entry_file": "checkpoint.md"
  }
}
```

这就像进程调用 `sleep_until(event, resume_context)`——不是裸睡，是带着恢复说明睡觉。

Harness 有最终裁决权。Agent 提议，Harness 根据成本降级。程序请求 syscall，操作系统决定怎么调度。

---

## 所有 Trigger 都是中断

在操作系统里，键盘输入、磁盘完成、网络包到达——全部统一为中断。内核不关心中断的来源，它只查中断向量表，跑对应的 handler。

Agent Runtime 也一样：

```
UserMessage       → 用户敲键盘
ToolFinished      → 磁盘读写完成
TimerExpired      → 时钟中断
WebhookReceived   → 网络包到达
HumanApproved     → 特权指令
AgentFinished     → 子进程退出
```

从 Harness 视角看，它们长一样：

```
Trigger { session_id, type, payload }
```

没有区别。统一调度。这是 Harness 能做内核的前提。

---

## 最终形态

没有再往前了。Agent Runtime = 操作系统 就是这个方向的终局。

```
Agent ≠ 模型
Agent ≠ 进程
Agent ≠ 容器
Agent = Session + Harness(Model, Tool?, SubAgent?, ...)
     = 持久状态 + 参数化调度器

其中：
  Model = CPU（必需）
  Tool = 设备驱动（可选）
  SubAgent = 用户进程（可选）
```

---

## 原文其实在讲同一件事

Anthropic 那篇《Scaling Managed Agents: Decoupling the brain from the hands》绕了一大圈，最后想说的其实是：

**Agent 的思考循环和执行环境应该像 CPU 和外设一样解耦。Session 是资产，Harness 可以销毁。**

他们没有直接说"Agent Runtime = 操作系统"，但每一个设计决策——Session 持久化、Harness 可销毁、Sandbox 远程化、事件驱动唤醒——都是操作系统常识在 Agent 世界的必然投影。

操作系统更准确。
