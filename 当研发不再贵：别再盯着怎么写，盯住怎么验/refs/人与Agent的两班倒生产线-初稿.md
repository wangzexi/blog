# 2027 的研发流程：人与 Agent 的两班倒生产线

过去这些年，研发流程里那些排期、看板、评审、同步会、站会，看起来越来越重，很多人因此很容易得出一个结论：流程太多了，应该砍掉。

但这个判断不够准确。过去那套流程之所以成立，不是因为企业天然喜欢复杂流程，而是因为软件生产的执行主体主要是人。人有上下班时间，有注意力上限，有沟通切换成本，也会在跨团队协作里不断丢上下文。所以企业必须围绕人类执行成本去设计流程，尽量保护昂贵的人类工作时间。排期是为了避免资源冲突，看板是为了让长期任务可视化，评审是为了减少返工，同步机制是为了让多人协作不要失控。

旧流程没有错。它只是服务于一个旧前提：任务的推进速度，基本等于人类的推进速度。

现在这个前提开始松动了。

## 一、真正变化的，不是 AI 比人快一点，而是任务第一次可以跨夜运行

很多人讨论 Agent 时，容易把重点放在“它是不是比工程师更强”。这当然重要，但它并不是流程变化最关键的起点。

更关键的变化是，任务第一次可以在无人值守时继续推进。

就算 Agent 单步执行不比最强的人类更快，只要它能连续跑，白天接任务，晚上继续执行，第二天早上把中间结果、失败记录、候选方案和待确认问题一起交回来，任务就已经不再严格受人类班次约束了。

这件事会带来一个非常实在的后果：很多执行层工作不再需要被切成完全适配人类协作节奏的碎片。过去你必须安排谁今天做什么、明天接什么、卡住了去哪同步，是因为晚上没人继续干。现在 Agent 可以在夜里继续试、继续改、继续验证、继续失败、继续重试。只要任务边界足够清楚，很多工作会从“等人排班接力”变成“让系统持续运行”。

而且这还不是一个短期现象。并不是所有任务都需要最强模型，模型成本也还会继续下降。一旦无人值守的连续执行足够便宜，很多原本被认为“需要精细排期”的任务，成本结构都会变。

所以流程变化的起点，不是 AI 比人快一点，而是任务第一次可以脱离人类班次连续推进。

## 二、未来更自然的最小工作单元，可能不是任务单，而是任务群

如果任务可以持续运行，那一张静态任务单就不太够用了。

任务单擅长描述目标，却不擅长承载一个任务在执行过程中的持续变化。真实任务需要的不只是“做什么”，还包括补充信息、上下文修正、执行状态、阻塞、阶段性结果、验收判断、继续推进的条件。这些东西不是一次性填完的，而是在推进过程中不断长出来的。

所以未来更自然的最小工作单元，可能不是一张静态卡片，而是一个持续运行的上下文容器。这个容器里同时存在目标、讨论、执行、反馈和验收。换句话说，不是“一个任务对应一个字段集合”，而是“一个任务对应一个持续活着的上下文”。

在企业现实里，这个容器最自然的形态很可能就是任务群。

原因并不神秘。企业真实的信息流中心本来就在 IM。无论审批系统、工单系统、文档系统还是数据报表系统，最后大量提醒、追问、协调、转发和链接分发都会回到群里或者私聊里。既然信息天然流回 IM，任务最自然的运行位置也会越来越靠近 IM，而不是再额外造一个要求所有人主动进入的新平台。

所谓“一个任务一个群”，本质上不是聊天软件偏好，而是在说：一个任务需要一个持续共享上下文的容器。

## 三、Agent 留在任务中心持续执行，人类只在关键节点介入

一旦任务群成为默认容器，人和 Agent 的分工也会跟着重排。

过去是人推进任务，系统辅助记录。以后更可能变成 Agent 持续推进任务，人类只在关键节点介入。

这里的关键不是“Agent 管理所有人”，而是大多数执行动作会围绕 Agent 的运行节奏展开。Agent 可以接收任务、分解步骤、调用工具、修改文件、跑测试、提交结果、汇报异常。普通执行角色并不会消失，但他们不再需要承担所有连续推进动作，而是更多地在关键节点补信息、做判断、做验收。

这三类人类职责会变得更突出：

- 补充信息：目标是什么，边界是什么，哪些约束不能碰
- 做判断：多个候选方案里选哪个，哪些风险可接受，哪些要回滚
- 做验收：结果是否满足要求，是否可以放量，是否可以发布

这其实意味着组织节奏在变化。过去系统的主要职责是记录人做了什么，以后系统的主要职责会变成让 Agent 持续跑，让人类在少数高价值时刻介入。

从这个角度看，所谓“人与 Agent 的两班倒”并不是一句修辞，而是一种新的生产节奏：人白天定义、校正和验收，Agent 在更长的时间窗口里持续执行。

## 四、动态任务流会留在 IM，执行事实会沉淀到代码库

如果任务群跑在 IM 里，一个很自然的问题就来了：那是不是所有东西都应该留在 IM？

不是。

IM 适合承载动态过程，不适合承载长期事实。

动态过程包括：

- 临时追问
- 状态变化
- 阻塞反馈
- 协调信息
- 提醒和催办

这些天然适合在 IM 里流动，因为它们本来就是高频、临时、需要被及时消费的信息。

但和研发执行直接相关的事实，不应该继续散落在聊天记录里，而应该尽量沉淀到代码库或与代码库等价的事实系统里，比如代码、配置、Prompt、测试、benchmark、实验结论、发布边界和回滚策略。

也就是说，未来更合理的信息结构不是所有东西都塞进同一个平台，而是把不同类型的信息放到最适合的位置：

- 原始任务流留在 IM
- 原始实现事实进入代码库
- 看板、统计、聚合、分析这些视图由系统自动派生

过去很多组织的问题是让人反复在不同平台搬运同一份信息。群里说一遍，文档里写一遍，看板里再填一遍，周报里再抄一遍。Agent 真正带来的变化，不只是替人执行，更是可以把这些派生视图自动生成出来，让人不再负责重复录入。

所以以后看板不是一定消失，而更可能从“原始录入系统”变成“派生观察视图”。

## 五、看板不会立刻消失，但它不再应该靠人手工维护

很多人会追问一句：如果都在 IM 里推进了，那看板是不是就没必要了？

答案是分情况。

对于短周期、边界清晰、可快速验证的任务，IM 加 Agent 的执行反馈可能已经够用。因为任务本身推进得足够快，人类真正关心的是结果和异常，而不是每天的人肉状态更新。

但对于跨团队、长周期、依赖复杂的任务，可视化视图依然有价值。问题不在于看板本身，而在于看板过去之所以令人厌烦，是因为它常常要求人类手工维护，把本来已经在群里、代码库里存在过的信息再复制一遍。

未来更合理的做法是：

- 群里保留任务的动态过程
- 代码库保留任务的执行事实
- 看板从这两类原始信息自动派生

这样看板就不再是一个“大家必须去填”的地方，而是一个“系统自动生成给你看”的地方。

这也是为什么我不太相信未来会再冒出一个巨大的“总平台”统一承接一切。更可能的路径是，任务继续在 IM 中心运行，事实继续在代码库中沉淀，结构化观察视图由 Agent 和基建自动拼出来。

## 六、未来研发不会消失，但会变成产线维护者和质量责任人

当产品、设计师、运营，甚至部分用户都可以更直接地提出需求、生成原型、推动试验，很多人会自然问：那研发还剩什么？

答案不是“研发依然亲手做每一个需求”，而是研发会越来越像一条软件产线的维护者和质量责任人。

因为真正稀缺的东西会变成这些：

- 环境是否稳定
- 工具链是否可用
- 权限和资源边界是否安全
- 测试和发布机制是否足够可靠
- 自动产出的结果是否可验证、可回滚、可审计

当生成能力越来越强，系统里会出现大量候选实现、临时分支、实验版本、短命环境和自动化产物。谁来保证这些东西不会把生产环境拖垮，谁来定义什么能上线、什么必须回滚、什么需要人工复核，谁来维护这条产线本身，这些问题最后都会重新落到研发身上。

所以未来研发的核心价值，不再只是写出功能，而是维护一条能让更多人安全产出软件的自动化生产线。

这不是研发退场，而是研发角色上移。以前研发更像亲自施工的人，未来研发会越来越像维护施工系统、验收质量边界、处理复杂异常的人。

## 结语

如果把这件事讲到最简，未来研发流程的变化其实只有一条主线：

过去流程围绕人类执行成本设计；现在任务可以由 Agent 跨夜连续运行，所以最小工作单元会从任务单转向任务群，动态任务流留在 IM，执行事实沉淀到代码库，人类退到补信息、做判断、做验收这些关键节点，研发最终转型为产线维护者和质量责任人。

这条线成立以后，排期、看板、评审、同步都不会立刻消失，但它们会被重新安排位置。旧时代那些为了保护人类执行时间而存在的流程，会逐步变成围绕系统连续执行能力重新设计的流程。

未来最重要的变化，也许不是“AI 会不会写代码”，而是软件生产第一次拥有了一条真正可以两班倒运转的执行系统。
