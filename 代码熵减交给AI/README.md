# 把代码熵减交给 AI：每天产出一个可 review 的小 PR

普通 AI 辅助编码通常是人先发现问题，再叫 AI 去改。

这次反过来：每天上午 AI 自己找一个直播交易代码里的小熵减点，改完后提交 Draft PR，再 @ 同学 review。

这次已经落地了三类东西：

- 3 个真实熵减 PR
- 3 个周期任务：每天 09:00 自动产出熵减 PR；每晚 19:00 提醒 CR；每周四 10:00 提醒可跟车合入的 PR
- 1 套直播交易代码熵减工作流：`mlive-entropy-reduction` Skill、配套脚本、动态评审组、核心组件测试包和双 case 冒烟规则

所以这件事不是“AI 帮忙改了几行代码”，而是把原本靠人自觉的治理类脏活，变成每天自动发生、团队只需要轻量 review 的小 PR。

## 一、我们要解决什么

直播间 C 端交易代码长期迭代，代码里会自然积累一些小复杂度。

这些问题通常不是事故级问题，但会持续增加维护成本：

- 无用转接层、只透传的包装层、空样板代码
- 可以 early return 的嵌套逻辑、只调用一次但增加跳转成本的函数
- 过期注释、typo、命名不统一
- 重复判断、重复参数拼装

目标不是大重构，而是借每周 QA 回归的节奏，每天做一个小的、低风险熵减 PR，把维护成本一点点降下来。

约束很明确：

- 不做大重构，不改核心业务语义
- 不碰接口参数、埋点、缓存 key 等隐式契约
- PR 要认知一致：允许多文件，但必须是同类问题
- 先提 Draft PR：标题、分支、base、描述和测试方法都按规范补齐

这里的核心判断是：

> AI 最适合先接手那些重复性的治理类脏活：高频、低风险、有规则、可验证。


## 二、AI 实际提出的几个熵减 PR

| PR | 仓库 | 变更点 | 为什么适合作为熵减 |
| --- | --- | --- | --- |
| `[熵减] 简化 CommonGoodsShelf 货架容器组件` `/内部链接_已脱敏/mlive-packages/pr/4950` | `ssa/mlive-packages` | 删除 `packages/CommonGoodsShelf/src/GoodsShelfContainer.tsx` 里的空 `Props` / `State` 和空 constructor | 低风险、行为不变，减少后续阅读噪音 |
| `[熵减] 简化自建商详单元素样式数组` `/内部链接_已脱敏/mlive-hotel/pr/1605` | `ssa/mlive-hotel` | 将自建商详里的 `style={[styles.xxx]}` 改成 `style={styles.xxx}` | 去掉无语义的单元素数组构造，行为不变，也少了一次不必要的数组创建 |
| `[熵减] 修正订单页返回按钮函数拼写` `/内部链接_已脱敏/mlive-hotel/pr/1606` | `ssa/mlive-hotel` | 将订单页里的 `renderBackButon` 修正为 `renderBackButton`，并同步调用点 | 修正 typo，方便后续检索和维护 |

其中第一个货架 PR 最有代表性。

重点看第一个 PR：它暴露了 AI 做熵减时最容易忽略的边界。

AI 第一版方案是把 class 组件改成函数组件。看起来更简单，但人类 review 提醒：class 组件的 `ref` 问题要关注。

AI 继续调研后确认：虽然仓库内当前没有显式传 `ref`，但它是公共导出组件，上游 lazy 机制也可能透传 `ref`，直接改成函数组件不够保守。

最终 PR 改成更保守的版本：

- 保留 `export class GoodsShelfContainer extends React.PureComponent`
- 删除空 `Props` / `State` 和空 constructor
- 保留公共导出和潜在 ref 语义

这个 PR 很小，但关键点在于：AI 不能只追求把代码写得更现代，还要结合当前工程里的导出方式、加载链路和历史用法，识别公共导出、`ref` 语义这类隐式契约。

这个 review 关注点后来也被记录进 `mlive-entropy-reduction`。未来再做类似熵减时，AI 会默认关注公共导出组件、class / function 形态变化和潜在 `ref` 语义。


## 三、我们怎么让它每天自动跑起来

### 1. 把规则写成 Skill

我们新增了 `mlive-entropy-reduction` Skill，记录稳定上下文：

- 目标和范围：日常熵减目标、直播交易代码关注范围
- 编码和 PR 粒度：编码原则、PR 认知一致要求
- 风险和验证：隐式契约风险点、测试包和冒烟要求
- 协作流程：动态 reviewer、PR 评论处理、晚间 CR 和周四 Merge 提醒

长期任务不能依赖一次聊天里的记忆。Skill 的作用，是让后续 Agent 即使没有完整上下文，也能按同一套标准继续执行。

这里还有一个重要机制：人类 review 评论会进入持续记忆。

处理 PR 评论时，AI 不会直接照单全收。它会先读评论上下文和相关代码，再判断评论是否成立。如果成立，就改代码、回复评论、标记解决；如果不成立，就在 PR 里说明判断依据。

更关键的是：有复用价值的评论会沉淀回 `mlive-entropy-reduction`，变成后续熵减任务的审查注意点。

也就是说，人类 review 不只是修正当前 PR，它会持续改变 AI 下一次看代码的方式。

### 2. 做成周期任务

我们创建了三个周期任务：

- 每天 09:00：做一次日常熵减 PR，通过检查后在群里同步 PR，并 @ 人 review
- 每晚 19:00：检查 `[熵减]` open PR、未解决评论和 CR 状态，必要时提醒当前群成员 review
- 每周四 10:00：比较最新 release diff，结合依赖图推导本周实际发布 bundle，提醒可跟车合入的熵减 PR

这是它和普通 AI 辅助编码最大的区别。

普通模式是人发起任务，AI 响应；这里是 AI 到点自动启动，把选点、改代码、验证、建 PR、补证据一路推进到可 review 状态。人不需要每天想“今天让 AI 优化哪里”，只需要处理一个已经收敛过的小 PR。

群内同步被压缩得很短，只发 PR 名称、diff 链接和 review @：

```text
[熵减] PR 标题：PR diff 链接 @两位同学 麻烦 review 下
```

核心信息不堆在群里，而是放进 PR：

- 变更说明：原因、内容、风险判断
- 本地检查：`lint`、`tscheck`、测试方法
- 运行态证据：测试包二维码、进房链接
- 冒烟结论：iOS 截图、无 RN 红屏结论

### 3. 把重复动作脚本化

周期任务能让事情持续发生，但里面仍然有很多机械操作：创建 PR、同步 reviewer、跑校验、评论证据、审计 PR、按 release diff 判断哪些 PR 可以跟车合入。

这些动作如果每次都让 AI 临场发挥，执行成本高，也容易在细节上波动。所以进一步把确定性动作脚本化：

- `create_entropy_pr.ts`：创建 Draft PR，处理 base、reviewers 和 PR 描述
- `run_entropy_validation.ts`：统一跑 `lint` / `tscheck`
- `run_goods_shelf_smoke.ts` / `comment_bundle_evidence.ts`：跑货架双 case 冒烟，并把证据评论到 PR
- `audit_entropy_prs.ts`：检查 Draft、分支名、base、reviewers、描述完整性
- `report_pending_entropy_reviews.ts`：生成晚间 CR 提醒
- `report_release_diff_entropy_prs.ts`：按 release diff 和 bundle 交集生成周四 Merge 提醒

形成的分工是：

```text
脚本负责确定性动作
AI 负责判断型决策
人类负责风险确认和最终 review
```

脚本化主要降低执行成本；选点、风险判断、截图判断和评论处理，仍然交给 AI 与人类 review。


## 四、常见问题

### 1. AI 的 PR 产能是越多越好吗？

AI 一天可以提出很多 PR，但团队不一定消费得了。

这次我们选择每天一个：当天给团队 review，一周就是五个，刚好可以进入本周 QA 回归视野。

这里不能贪多。AI 如果一次抛出太多 PR，很快就会让 reviewer 疲劳，最后大家不愿意花时间看，这套机制也会被废弃。

治理类工作要细水长流：每天一个小 PR，让团队能稳定接住。

### 2. AI 把流程跑完后，人类 review 看什么？

当 AI 已经补好 PR 描述、跑完校验、打出测试包、上传截图后，人类 review 不应该再主要检查流程有没有走完，而是看这些判断是否成立：

- 这个改动是否真的必要
- 是否破坏隐式契约
- 是否符合当前业务风险
- 验证证据是否可信

对直播交易代码来说，`lint` 和 `tscheck` 不够。

货架、讲解气泡商卡这类核心组件，还需要打测试包并做 iOS 冒烟。

PR #4950 后续补了完整验证证据：

- 测试包：`rn_hotel_hotel-mlive 0.2.1040`，附 Talos 日志和 Pillow 锁包页
- iOS 模拟器锁包进入直播间，截图确认进入货架列表
- 截图里能看到锁包版本，无 RN 红屏
- 测试计划、二维码、链接和截图都评论到 PR，方便 reviewer 进一步验证

后续这类机械化验证也应该尽量先让 AI 自己做掉：能跑的检查先跑，能补的证据先补，能初步判断的截图先判断，再把结果交给 reviewer 复核。

还有一个规则很重要：

> 截图不是为了交差，AI 自己也要看截图。

AI 不能只把截图贴到 PR 里，还要自己判断页面有没有红屏、空白，以及是否真的进入了目标页面。这个要求需要明确写进流程里，因为 AI 真的可能只完成“截图”这个动作，而没有认真看图。

### 3. 哪些事情必须由人来做？

这次实践里，我们没有让 AI 自动 merge，也没有绕过 review。

合理边界应该是：

```text
AI 可以提出变更
AI 可以执行验证
AI 可以整理证据
AI 可以推动 review
但最终合并，以及测试仍然需要人类
```

原因很简单：AI 可以执行流程和提供判断依据，但人才是责任主体。

### 4. 熵减 PR 应该长什么样？

熵减 PR 的目标不是展示 AI 能改多少代码，而是让人类能轻松 review。

治理类工作短期收益不明显，如果一个 PR 读起来很累，reviewer 很快就会失去耐心。所以它应该尽量满足一个标准：人类最好能在 60 秒内看懂问题模式、风险边界和验证结论。

`[熵减] 简化自建商详单元素样式数组` `/内部链接_已脱敏/mlive-hotel/pr/1605` 是一个例子。它改了 7 个业务文件里的 8 处位置，但问题模式完全一致：都是把无语义的单元素 `style` 数组从 `style={[styles.xxx]}` 改成 `style={styles.xxx}`。

所以 PR 粒度应该是“认知一致”，不是“diff 小”。一个小的、清楚的 review，也适合放在一天开始时作为进入工作状态的入口。

### 5. 信息应该集中在哪里？

群聊只做通知：PR 名称、PR diff 链接和 review @。

变更原因、验证结果、二维码、截图、风险判断和评论处理记录，都应该留在 PR 里。群聊适合提醒，不适合承载审查上下文。

review 评论不只影响当前 PR。只要评论有复用价值，就应该沉淀回 Skill，变成下一轮 AI 的默认检查项。

### 6. 哪些治理杂活适合周期化？

适合周期化的，不一定只有代码熵减。

关键是这类事情要满足几个条件：上下文稳定、任务具体、动作机械、结果可验证。

比如删死代码、修 typo、清理过期注释、简化一次性函数、补齐 PR 描述和测试方法。这些事情过去靠人自觉，现在可以让 AI 定期捡起来做。

AI 落地不一定要从大项目开始。更现实的切入点，是把团队里那些高频、低风险、可验证、但过去总靠人自觉的小事，变成每天自动发生的事。
