# 2027 的研发流程：讨论记录

## 这份记录在回答什么

这份记录不是正文，而是把刚才反复确认的判断、边界、反例和附录方向先沉淀下来，后续写文章时可以直接从这里取材。

文章要回答的问题很简单：

- 如果 Agent 可以跨夜连续执行，研发流程会怎么变
- 为什么未来更自然的工作单元可能是任务群，而不是任务单
- 为什么 IM 会变成任务运行中心
- 为什么研发不会消失，而是转成产线维护者和质量责任人

贯穿全文的主线也已经比较清楚：

过去流程围绕人类执行成本设计；现在任务可以由 Agent 跨夜连续运行，所以最小工作单元会从静态任务单，转向持续运行的任务群。动态任务流留在 IM，执行事实沉淀到代码库，人类退到补信息、做判断、做验收这些关键节点，研发最终转型为产线维护者和质量责任人。

## 一、旧流程为什么曾经是对的

一个容易写偏的地方，是把过去的排期、看板、评审写成“官僚主义”。这个角度不对。

更准确的说法是：

- 过去软件生产的执行主体主要是人
- 人有上班时间、切换成本、沟通成本、注意力上限
- 人类执行很贵，所以流程天然围绕“保护人类执行时间”展开
- 排期、看板、评审、同步，本质上都是为了让昂贵的人类协作尽量不要失控

所以旧流程没有错。它只是服务于旧的成本结构。

这一点很重要，因为它决定了全文不是“旧流程批判文”，而是“成本结构变化后的流程重组”。

一句可以反复使用的话是：

旧流程没有错，只是它服务的执行成本结构开始变化了。

## 二、真正的转折不是 AI 更聪明，而是任务第一次可以跨夜运行

这里最关键的，不是强调 Agent 每一步一定比人快，而是强调任务第一次可以脱离人的班次连续推进。

几个支撑点已经比较明确：

- Agent 可以在晚上继续跑
- 不是所有任务都需要最强模型
- 模型成本还会继续往下掉
- 只要无人值守运行成立，大量任务的连续执行成本就会显著低于人类

所以文章里的核心判断应该是：

流程变化的起点，不是 AI 比人快一点，而是任务第一次可以脱离人类班次连续推进。

这个判断会自然推导出后面的所有变化。因为一旦执行不再跟着人的班次停机，很多用来组织人类接力的流程结构都会被重写。

## 三、最小工作单元为什么会从任务单变成任务群

这里的关键区别是：

- 任务单是静态描述
- 任务群是持续运行的上下文容器

真正的任务执行并不只需要一句需求描述。它需要：

- 目标
- 补充信息
- 执行状态
- 阻塞
- 中间决策
- 结果
- 验收结论

这些东西不是一次性写完的，而是持续流动的。所以未来更自然的最小工作单元，可能不是一张静态卡片，而是一个持续运行的上下文容器。这个容器在企业里最自然的形态，就是“一个任务一个群”。

这里要避免把“群”写成产品形态本身，重点不是聊天软件，而是“持续共享上下文”的容器。之所以会落在 IM，是因为企业现实里信息流中心本来就在 IM。

可以直接写成：

- 企业里大量提醒、追问、同步、链接分发最后都会回到 IM
- 所以任务最自然的运行位置，本来就在 IM
- Agent 不需要强行把任务搬去另一个新平台，它更可能直接嵌入已经存在的信息流中心

## 四、人和 Agent 的关系不是“谁替代谁”，而是执行节奏重排

这里不要写成“Agent 管理所有人”。这个说法会让人本能反感，而且也不准确。

更准确的表达是：

- Agent 承担大部分连续执行
- 普通执行角色和 Agent 围绕同一个任务中心协作
- 人类主要负责高价值介入节点

这些高价值介入节点大致就是三类：

- 补充信息
- 做判断
- 做验收

如果要把这段说得更有画面，可以这么理解：

过去是人推进任务，系统辅助记录。
以后是 Agent 持续跑，人类按需介入。

也就是说，真正变化的不是“有没有人”，而是谁在持续推进任务，谁在关键节点做裁决。

## 五、为什么任务流会留在 IM，而执行事实要沉淀到代码库

这里要把“信息流”和“事实流”分开写。

动态任务过程更适合留在 IM，因为那里承载的是：

- 追问
- 状态更新
- 临时补充
- 阻塞反馈
- 协调和提醒

但跟研发执行直接相关的事实，不应该继续散落在聊天记录里，而应该尽量沉淀到代码库或与代码库等价的事实系统里，比如：

- 代码
- 配置
- prompt
- 测试
- benchmark
- 实验结果
- 发布边界
- 回滚策略

所以比较稳的表达不是“IM 跑任务，代码库存事实”这种口号，而是：

- 原始任务流留在 IM
- 原始实现事实进入代码库
- 看板、统计、聚合、分析都应该是派生视图，而不是让人重复录入的原始系统

这里还可以顺手限定讨论范围，避免文章被误读成“企业所有管理信息都放聊天软件里”。更准确的边界是：

这里讨论的是研发执行层的信息组织，不是企业全部管理活动。

## 六、研发不会消失，但角色会向产线维护者和质量责任人转移

这个结论必须回答一个读者一定会问的问题：

如果产品、设计师、运营，甚至部分用户都能更直接地产出原型和实现候选，那研发还负责什么？

答案不是“研发依然负责写所有功能”，而是：

- 维护产线
- 维护环境
- 维护测试和发布边界
- 维护权限与资源约束
- 对产线产出的质量负责

所以未来研发更像：

- 自动化生产线的维护者
- 规则制定者
- 质量责任人
- 复杂问题的最后兜底者

这不是研发价值下降，反而是研发从“亲手搬运每个需求”转向“维护一条能稳定产出软件的系统”。

一句适合作为收束的话是：

未来研发的核心价值，不再只是写出功能，而是维护一条能让更多人安全产出软件的自动化生产线。

## 七、文章里需要明确写出的几个边界

为了避免论证被人抓住极端案例，正文里至少要明确这些边界。

### 1. 不是所有任务都适合完全自动化

短周期、局部封闭、可验证的任务最容易进入自动闭环。

跨团队、依赖复杂、审批密集、对外部环境高度敏感的任务，依然需要更多人类协调。

### 2. 看板不会立刻消失，但会从原始录入系统变成派生视图

对短周期任务，IM 加 Agent 反馈可能足够。

对跨团队和长周期任务，看板仍然可能存在，但更合理的状态是：

- 看板不再由人手工维护
- 看板由 Agent 从任务流和事实流里自动生成

### 3. 一个任务一个群会有群膨胀问题

这是必须正面承认的质疑点。可行的解法不是回到旧平台，而是让 Agent 负责：

- 自动归档
- 群索引
- 通知精简
- 订阅机制
- 群内摘要

### 4. IM 是任务中心，不等于 IM 是唯一系统

IM 适合承载动态过程，但结构化视图、权限控制、代码事实、实验数据仍然需要各自最适合的系统承载。只是这些系统不应该要求人类重复搬运信息。

### 5. 这不是“AI 替代所有人”

更准确的说法是：

- 大量连续执行会被 Agent 吞掉
- 人类会退到更少但更关键的节点
- 组织结构和岗位边界会重排
- 责任不会消失，只会重新集中到环境、规则和质量边界上

## 九、建议的正文写法

正文不要写成未来学宣言，也不要写成产品方案说明书。更适合的写法是：

- 先解释旧流程为什么成立
- 再指出成本结构哪里变了
- 再顺着推导组织形态、信息形态、岗位形态怎么变
- 最后回答读者最担心的质疑

语气上尽量克制，少用“颠覆”“重构一切”这种词，多用“更自然的最小单元”“更可能的默认形态”“会逐步转成”这类表达。

因为这篇文章真正有力的地方，不在于情绪，而在于逻辑链条很顺：

执行成本结构变了，流程就会变；流程变了，任务容器和信息容器就会变；容器变了，人和研发的角色就会跟着变。
