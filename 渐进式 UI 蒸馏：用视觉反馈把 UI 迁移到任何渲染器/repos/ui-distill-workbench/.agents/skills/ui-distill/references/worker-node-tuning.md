# Worker 单节点微调手册

这个文档给子代理使用。子代理只负责当前节点的数值收敛，不负责选择节点、不负责编排源端探针、不负责整页验收。

## 输入

主代理会提供：

- 当前节点名称，例如 `button`、`hero-card`、`title text`
- 当前节点的注释上下文，例如 `distill: hero`，以及具体 selector，例如 `.hero-title`
- 当前探针类型：`box` 或 `text`
- RN 参考截图路径
- HTML 截图输出路径
- 测量命令：盒模型用 `scripts/check-box.py`，文字用 `scripts/check-text.py`
- 允许修改的 HTML 选择器或行号范围

如果这些输入不完整，先停下汇报，不要猜节点。

## 写入边界

只修改 `pages/app.html` 中当前节点相关的 CSS。

不要修改：

- `../rn-new/src/App.tsx`
- `SKILL.md`
- 截图脚本
- 测量脚本
- 非当前节点的 HTML 结构

除非主代理明确授权，不要改变 DOM 层级。

## 基本循环

1. 截 HTML。
2. 跑当前探针类型对应的 checker。
3. 先读 checker 的核心指标，再判断建议是否适合当前结构。
4. 重复截图和 checker。
5. 记录最好结果。
6. 如果连续两步变差，回到最好结果并停止。

checker 建议是启发，不是命令。执行建议前先确认 probe 没有污染、祖先没有被隐藏、兄弟和子级透明规则正确，并且建议不会破坏布局上下文。

可以不执行 checker 建议的情况：

- 建议和主代理指定的探针规则冲突。
- 建议会修改当前节点之外的结构或视觉。
- 大文本偏差更像容器、line-height、padding、font fallback 问题，而不是 transform 问题。
- 局部 checker 变好但全局 diff 或相邻节点更差。
- 剩余误差更像渲染器上限。

不执行建议时，报告原因，并说明改用了哪个更符合结构的动作。

不要靠肉眼主调。截图只用于异常检查，例如品红污染、页面没刷新、截图尺寸错误。

主代理可能会从全局 diff 最大 cluster 派发当前节点。cluster 只表示目标来源；你的验收仍然看当前节点 probe 的 checker 结果，以及主代理最终计算的全局 diff 是否下降。

## 局部回滚

worker 必须维护当前节点的最好局部候选：

- 第一次 checker 结果是 baseline。
- 每次只做一个小改动。
- 改动后重新截图并运行 checker。
- 当前节点 checker 变好才作为候选保留。
- checker 变差就回滚到最好版本。
- 连续两次没有改善，停止当前节点。

worker 不判断整页是否最终成立。主代理会先恢复真色，再用全局 diff 做最终验收；如果局部变好但真色全局变差，主代理负责回滚该节点改动。worker 汇报时不要把局部 checker 改善描述成“最终通过”。

## 探针规则

- 盒模型探针：目标节点本身是 `#FF00FF`。
- 文字 box 探针：目标文本元素 `background: #FF00FF; color: transparent;`，只用于确认布局占位和锚点。
- 文字 glyph 探针：目标文字是 `#FF00FF`，目标容器背景恢复真色或透明。
- 非目标视觉应该已由主代理隐藏，但仍保留布局。
- 不要删除兄弟节点或子节点，因为删除会改变布局。
- 如果目标是某个子节点，不要对它的祖先使用 `opacity: 0`，否则目标也会一起不可见。
- 当前节点的祖先必须保留并可参与布局；当前节点的兄弟和子级才透明。

## 执行规则

- 一次只改一个变量。
- 优先考虑 checker 置信度最高且改动最小的建议，但必须先做结构判断；不要机械执行建议。
- 字体必须显式指定。遇到像字体 fallback 的误差，先检查 `font-family`，不要直接大幅改字号。
- 文本节点先修容器和布局占位，再修字体参数，最后才用 `transform` 做亚像素微补。
- 如果文本 `dx/dy/dw/dh` 超过 1px，不要优先用 `transform`；先检查父容器 padding、alignItems、margin、固定高度、line-height、inline/block 布局。
- 文本 background probe 只辅助定位布局盒，不作为最终字形标准；最终以 glyph probe 和全局 diff 为准。
- 不要把 `scaleX/scaleY` 当作常规字体调参。只有主代理明确进入极限阶段，并且正常字体参数、容器参数和 section 级 y 对齐都已经验证过，才允许做 0.5%-1% 的微缩放实验。
- 微缩放必须单项验证，再由主代理组合验证。单个节点 checker 变好不代表可以落地。
- 对重复结构，先把第一个实例收敛，再用同样参数验证其余实例；只有测量变差时才单独微调。
- 盒模型 bbox 已经完全重合时，不要继续为了极小 XOR 长时间调参。
- 文字 bbox 接近后，最多试少量 `font-size`、`line-height`、`font-weight` 组合；如果 XOR 不稳定下降，记录为字体渲染差异。
- 局部 probe 改善必须等待主代理真色全局验证；真实颜色、抗锯齿、阴影、相邻节点和父级裁剪可能让全局 loss 反向变差。
- 当 bbox、ink area 或 density 已接近，但连续两次小步实验都不降低 checker loss，停止当前节点，记录为疑似渲染上限。
- 不要为了追逐单像素 XOR 引入影响兄弟节点或父级布局的改动。

## 汇报格式

完成后只汇报高信号信息：

- 当前节点和探针类型
- distill 注释上下文和 selector
- 最好 checker 摘要：核心指标
- 执行过的 checker 建议
- 修改过的 CSS 参数
- 停止原因
- 是否有疑似污染或结构问题
