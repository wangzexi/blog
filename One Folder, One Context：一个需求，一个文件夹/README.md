# One Folder, One Context：一个需求，一个文件夹

我们认为，参与一个需求的所有人，应该在同一个文件夹下工作。

```text
需求名/
├── PM/
│   └── README.md
├── DS/
│   └── README.md
├── RD/
│   └── README.md
└── QA/
    └── README.md
```


## 背景：信息存在，但零散

我们总是四处找信息：

```markdown
- PRD、会议纪要、技术方案在在线文档里
- 设计稿在设计工具里
- 测试用例在用例系统里
- 竞品调研、SQL 片段、Excel 散落在本地文件夹里
- …
```

汇聚到粘到文件夹，再让 AI 阅读、写初版、完善，最后粘回在线文档…

那为什么不直接就保持在文件夹？

文件夹就是计算机诞生 60 多年来，所有人最熟悉的信息组织方式，现在的 AI 也都非常擅长在文件夹里工作，而且天然支持多媒体。

如果一个需求的一切都在一个文件夹里，那对人、对 AI 都会非常方便。


## 文件夹结构：延续当前单周迭代的组织方式

```text
单周迭代/
└── 2026/
    └── W15-04-06/
        └── 需求名/
            ├── PM/
            │   └── README.md
            ├── DS/
            │   └── README.md
            ├── RD/
            │   └── README.md
            └── QA/
                └── README.md
```

也就是：

- 每周一个迭代，比如 `W15-04-06`。
- 一个需求一个文件夹。
- 每个角色在自己的文件夹里写。
- 每个角色的 `README.md` 是核心文档，比如 PM 是 PRD，RD 是技术方案，DS 是设计稿，QA 是测试用例。
- 其它过程资料也可以自由放在角色文件夹里，比如截图、调研、SQL 分析、图表、语音转录、会议纪要等。


## Q&A

### 就只是把文档换个地方放？

是的。

就是从在线文档，换到本地文件夹的 Markdown。

同一个需求相关的信息放得越近，人和 AI 理解它的成本就越低。

PRD、设计、技术方案、测试用例、代码和过程材料放在同一个文件夹里，就可以被一起读取、追溯和版本化。

### 大家的文件夹怎么互相同步？

Git。

Git 仓库就是一个文件夹，每个人都可以很轻易克隆、修改、提交和上传。

这篇文档就在团队内的 Git 仓库 context-repo。

本质上，我们只是需要一个有文件版本历史、方便多人同步、AI 也熟悉的文件管理工具。Git 刚好就是，也确实被全球工程师验证很好用。


### 让产品经理和设计师用 Git？

是的。

事实上，大家只需学会两个动作：保存（Commit），上传（Push）。

> 你可以很轻易让 AI 克隆这个仓库到电脑桌面，然后让它在 [来试试Git](./来试试Git/README.md) 文件夹里，帮你写个打油诗～

因为大家都在工作在自己的文件夹，不会互相冲突，也就不需要研究什么分支、PR。

真正需要形成习惯的是：相关材料尽量放回对应需求文件夹。

### 大家都往需求文件夹里写会不会很乱？

不会。

因为文件夹骨架是稳定的。

每个角色都有自己的文件夹，也约定了固定入口：`README.md`。其它过程材料再怎么自由摆放，都不会影响阅读。

这套方式的重点不是把文件夹整理得很漂亮，而是让同一个需求的核心上下文不要散。

### 这是不是要求大家马上变成全栈？

不是。

全栈协作的第一步，不是让每个人立刻跨角色做事，而是先让信息在一起。

也就是先把大家工作的物理信息边界稍微往外扩一圈：产品能看到设计、研发、QA 的上下文，设计能看到产品背景和研发约束，研发和 QA 也能看到前面的决策过程。

思想和习惯的转变需要时间，但文件夹可以先把空间准备好。大家仍然在自己的角色文件夹里工作，只是同一个需求的上下文不再被系统和个人文件夹切碎。

### 现有的 需求管理系统、迭代表格、周报等怎么办？

不变。

需求文件夹是单一信息源，在线文档、需求管理系统、迭代表格等外围系统可以继续作为视图、索引和元数据。

如果外围系统需要同步，就提供 Skill，在更新时把需求文件夹里的信息同步出去。

### 这对产品、设计、研发、QA 有什么好处？

产品写需求时，可以把调研、数据、取数 SQL、会议纪要、截图和 PRD 放在一起，后续不用反复解释背景。

设计可以直接读取产品背景和历史材料，产出的说明、静态稿、交互补充也能留在同一个需求文件夹里。

研发可以直接看到需求背景、设计材料、测试关注点和历史决策，写技术方案或让 AI 辅助开发时不用重新拼上下文。

QA 可以更容易看到需求、设计、技术方案和代码变化，判断测试范围时不用到处拼上下文。

最关键的是，所有历史需求都会按同一套文件夹结构沉淀在一起。以后想追溯过去某个需求的产品背景、设计资料、技术方案或测试范围，可以直接问 AI，不用再靠人翻聊天记录和在线文档。

### 为什么会想到这个？

我们之前已经建立了业务统一索引仓库 [monorepo](./monorepo)，把前后端工程索引到同一个研发大文件夹里，解决的是研发侧看工程上下文的问题。

现在这个仓库继续往前走：把产品、设计、研发、QA 的需求上下文也放进来。
