# 从群聊到文件夹：AI 时代的组织信息设计

![文件夹 Runtime](./assets/folder-runtime-hero.png)

一个业务最怕的不是换人。

最怕的是：人一换，上下文也跟着没了。

为什么这个需求当时这么定？为什么这个字段不能改？为什么上次上线翻车？设计稿和 PRD 哪个才是新的？测试到底覆盖过哪些边界？这些东西往往不在一个地方，而是散在群聊、会议纪要、在线文档、个人文件夹和某几个老人的脑子里。

所以新人接手一个业务，第一件事不是工作，而是考古。

AI 进来以后，这个问题会更明显。人至少还能问同事、翻历史、靠经验猜；Agent 如果没有稳定上下文，就只能反复等人喂资料。每次启动一次 AI，就像重新招聘一个什么都不知道的新同事。

这就是我想解决的问题：

> 人可以换，执行器可以换，但业务上下文不能每次重建。

## 结论

AI 时代，组织需要把协作中心从“群聊”逐渐迁移到“文件夹”。

这里的文件夹不是普通资料夹，而是一个业务作用域的上下文容器。一个需求、一个模块、一个事故、一个客户项目，都可以是一个文件夹。

文件夹里放的不只是最终文档，还包括原始材料、历史记录、当前状态、判断规则、风险、待办，以及未来 Agent 可以执行的自动化任务。

简单说：

```text
文件夹 = 业务上下文
README = 对外状态接口
Agent = 可以驻守在文件夹里的执行器
```

传统组织架构图回答的是：谁负责什么。

这套文件夹结构回答的是：一个业务作用域的上下文在哪里，它如何被维护，谁可以读取它，谁可以继续执行它。

这不是在重画组织架构图，而是在重设计组织架构背后的上下文层。

## 一个需求一个文件夹

先看一个最小例子。

一个需求来了，我们不再让它散落在多个系统里，而是在一个 Git 仓库里建一个需求文件夹：

```text
需求名/
├── PM/
│   └── README.md
├── DS/
│   └── README.md
├── RD/
│   └── README.md
└── QA/
    └── README.md
```

PM 在里面放 PRD、调研、截图、会议转写。

设计在里面放设计说明、交互补充、取舍记录。

研发在里面放技术方案、接口草稿、风险判断。

QA 在里面放测试用例、回归范围、验收记录。

这件事第一步并不复杂：只是把同一个需求的上下文放到同一个地方。

它不要求大家立刻全栈，也不要求推翻学城、Ones、周报、IM。外围系统可以继续存在，只是逐渐变成视图和通知。真正的上下文源头，开始回到需求文件夹里。

对人来说，这减少了找信息的成本。

对 AI 来说，这意味着它终于有了一个可以长期读取的工作空间。

## 为什么不是群聊？

群聊适合沟通，但不适合保存状态。

群聊是一条事件流。它很适合提醒、讨论、临时决策、快速拉齐。但它天然会沉没，天然没有稳定结构，也很难告诉新人或 Agent：哪些消息已经过期，哪些才是当前真实状态。

一个需求往往会裂变出很多群：需求群、开发群、测试群、上线群、故障群。群很多，但真正稳定存在的不是这些聊天窗口，而是这个需求本身。

换句话说，群聊只是业务作用域的一种临时界面。

真正需要沉淀的是作用域的上下文。

文件夹刚好适合做这件事：它能放材料，能嵌套目录，能做权限边界，能被版本化，能被 AI 读取，也能让后来的人接着维护。

## 什么是业务上下文？

业务上下文不是一句背景介绍。

它包括：

- 这个业务为什么存在。
- 当前目标是什么。
- 历史上做过哪些决策。
- 哪些坑踩过。
- 哪些约束不能破。
- 上下游依赖谁。
- 什么情况算异常。
- 谁负责维护。
- 接下来要做什么。

过去这些上下文常常绑在具体人身上。

一个老同事知道为什么不能改某个字段，知道某个需求为什么妥协，知道某个合作方为什么难推进。这个人一走，组织丢掉的不只是一个执行者，而是一整块隐性上下文。

文件夹结构的价值，就是让业务上下文不要私有化地挂在人身上。

新人进来时，先进入这个文件夹。

Agent 进来时，也先进入这个文件夹。

人可以换，AI 可以换，工具可以换，但业务上下文持续存在。

## README 是接口

如果文件夹里什么都能放，它很快也会变成垃圾堆。

所以每个文件夹需要一个固定入口：`README.md`。

README 不需要写成大文档，它只需要回答几个问题：

- 这个目录当前在做什么？
- 现在状态是什么？
- 关键结论是什么？
- 有哪些风险？
- 谁负责？
- 外部如果要了解这里，应该先看哪里？

这样，一个文件夹就有了对外接口。

父目录可以读子目录 README，生成周报和风险汇总。

兄弟目录可以读彼此 README，了解依赖和状态。

Agent 也可以定时读 README，发现它和内部材料是否不一致。

目录内部可以很复杂，但对外要有清晰接口。

这就是文件夹能承载组织协作的关键。

## 文件夹会变成 Runtime

如果只有人使用文件夹，它主要是一个信息组织方式。

但当 Agent 具备定时任务、循环执行、工具调用和写回能力之后，文件夹就开始像一个运行环境。

Agent 可以每天醒来，读取某个目录：

- 看 README 有没有过期。
- 看需求有没有新增变更。
- 看设计、研发、QA 的状态是否对齐。
- 看风险是否被多次提到。
- 看哪些待办没人处理。
- 把结果写回文件夹，或者推送给负责人。

OpenAI Codex 的 Automations、Claude Code 的 scheduled tasks 和 `/loop`，都在说明一个趋势：Agent 不再只是被临时问一句的工具，它开始可以周期性地读取上下文、执行动作、写回结果。

这时，文件夹就不只是存储。

它变成了 Agent 可以长期驻守的业务运行空间。

所以我把它叫做文件夹 Runtime。

## 它为什么允许渐进式替换？

AI 替换人的前提，不是“模型突然变聪明”。

更现实的前提是：组织先把原来藏在人脑里的上下文外化出来。

一个岗位之所以难以被 AI 接管，往往不是因为每个动作都很难，而是因为动作背后有大量隐性判断：什么时候算异常，哪些历史坑不能再踩，出了问题找谁，哪些风险必须提前说。

如果这些东西都在文件夹里沉淀下来，替换就可以渐进发生：

```text
先让 Agent 观察
再让 Agent 汇总
再让 Agent 检查
再让 Agent 生成初稿
最后让 Agent 接管稳定流程
```

同样的逻辑也适用于换人。

当业务上下文在文件夹里，而不是绑在某个具体人身上，一个新人就能更快接手。业务不再因为某个人离开而上下文断裂。

所以这套结构支持两种替换：

```text
人类执行器 -> 新的人类执行器
人类执行器 -> AI Agent
```

真正稳定的不是执行器，而是业务上下文。

## 和 Context, not Control 的关系

这套思路和张一鸣讲的 **Context, not Control** 是同一个方向。

Context, not Control 的核心是：不要靠层层控制让组织运转，而是让更多人拿到充分上下文，基于上下文自己判断。

但在 AI 时代，我们还要继续往下问一步：

这些上下文到底放在哪里？

怎么保持更新？

怎么让新人能读懂？

怎么让 Agent 也能读懂？

怎么让上下文不只是帮助人决策，还能驱动自动化任务？

文件夹 Runtime 可以理解为 Context, not Control 的基础设施版本。

它不是增加控制，而是减少对具体人的依赖。通过把上下文建设好，让局部作用域里的人和 Agent 都能更自主地工作。

真正的原则不是不要控制，而是：

> 用最少的控制，维护最高质量的上下文。

## 怎么开始？

不用一步到位。

第一步，只要从一个需求开始。

建一个需求文件夹，让 PM、设计、研发、QA 把相关材料放进去。先不要追求完美结构，只要保证同一个需求的上下文不要散。

第二步，固定 README。

每个角色目录都要有 README，用来说明当前状态、关键结论、风险和下一步。过程材料可以自由放，但入口要稳定。

第三步，让 AI 做观察者。

先不要让 Agent 直接接管复杂工作。可以让它每周读一遍目录，提醒哪些 README 过期、哪些需求缺少测试信息、哪些风险反复出现。

第四步，把稳定工作写成规则。

比如测试前必须检查什么，需求变更后要提醒谁，哪些历史问题容易复发。规则写出来之后，Agent 才能反复执行。

第五步，让 Agent 驻守部分目录。

当一个作用域里的上下文足够稳定、规则足够清楚、异常边界足够明确，人类就可以逐渐退出重复执行，去处理新的、不稳定的、还没有被结构化的问题。

## 最后

这件事表面上是“一个需求一个文件夹”。

往深了看，它是在重新设计组织上下文。

过去组织依赖人记住上下文、传递上下文、解释上下文。

AI 时代，组织需要把上下文变成一种稳定资产：能被人读取，也能被 Agent 读取；能被新人继承，也能被自动化任务运行。

群聊仍然重要，但它更适合做通知和讨论。

真正长期存在的，应该是一个个业务作用域的文件夹。

人可以换。

执行器可以换。

但业务上下文要留下来。

当上下文被稳定沉淀，组织才可能真正进入 AI 原生的协作方式。
