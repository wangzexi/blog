# 从群聊到文件夹：AI 时代的组织信息设计

![文件夹 Runtime](./assets/folder-runtime-hero.png)

AI 时代，企业协作的中心会从“群聊”逐渐转向“文件夹”。

不是因为文件夹更先进，而是因为 AI Agent 的工作方式变了。

人类可以靠群聊工作。我们看到消息、理解上下文、记住历史、判断轻重缓急，再把散落的信息重新组织成行动。但 Agent 不适合长期生活在消息流里。消息会沉没，状态会断裂，历史不可继承，新的 Agent 每次加入都要重新喂上下文。

Agent 更需要一个稳定的状态空间：能读取当前状态，能追溯历史变化，能按规则定时醒来，能把结果写回去，能让下一个人或下一个 Agent 接着工作。

文件夹正好是这样的空间。

因为在一个业务里，真正长期稳定的不是某个人，也不是某个执行器，而是这个业务本身的上下文。人会换，团队会调整，工具会升级，Agent 也会一代一代替换；但一个业务的目标、约束、历史决策、上下游关系、风险模式和判断标准，会长期存在，并持续影响后续所有动作。

一个需求可以是一个文件夹。PM、设计、研发、QA 都在同一个需求文件夹下工作；调研、截图、会议转写、PRD、设计说明、技术方案、测试用例都放在里面。README 是这个文件夹对外暴露的状态接口，Git 负责版本历史，Agent 可以定时读取、汇总、更新。

第一步并不复杂：只是把同一个需求的上下文放到同一个地方。

但这一步背后，其实藏着一个更大的变化：组织不再只靠人和人之间的消息传递来维持运转，而开始把局部工作状态沉淀为一个个可读取、可追溯、可运行的文件夹。

未来 AI 原生的组织，可能会越来越像一棵持续运行的文件树。

## 先说结论

这套设计可以先用三句话理解：

**文件夹不是资料容器，而是工作作用域。** 一个需求、一个模块、一个事故、一个客户，都可以是一个文件夹。文件夹里不仅有最终文档，也有过程材料、历史变化、当前状态、参与者和后续动作。

**README 不是普通文档，而是状态接口。** 外部不需要读完整个文件夹，只需要先读 README，就能知道这个作用域当前是什么状态、谁负责、有什么风险、对外提供什么能力。

**Agent 不是被临时 @ 的工具，而是驻守在目录里的运行时协作者。** 它可以定时读取目录状态，发现变化，生成摘要，提出风险，写回结果。人类不再每次从头给 AI 解释背景，而是让 AI 长期生活在同一个上下文里。

所以，这不是一个“文档整理规范”。它更像是一种组织信息设计：

```text
文件夹 = 状态空间 + 原始事实 + 规则文档 + 自动化任务 + Agent
```

也可以更直白地说：这是一个“文件夹 Runtime”。

再往上说一层，它其实是在重新设计组织架构背后的上下文层。

传统组织架构图描述的是人和汇报关系：谁属于哪个团队，谁向谁汇报，谁负责哪个方向。但真正让组织能运转的，不只是这些线条，而是每个业务单元背后的上下文：目标、边界、历史、约束、依赖、风险、判断标准和工作方式。

过去这层上下文主要藏在人脑、会议、群聊、文档和流程里。现在有了 Agent，这层上下文需要变成 AI 可读取、可追溯、可更新、可定时运行的信息结构。

所以文件树不是在替代组织架构图，而是在给组织架构补上一层新的底座：

```text
组织架构图：谁负责什么
文件夹上下文：这个作用域如何持续运转
Agent Runtime：谁来定时读取、判断、更新和执行
```

这就是一种 AI 时代的组织上下文架构。

它还有一个更深的结果：一旦上下文和工作方式被稳定沉淀下来，执行器就可以被替换。

过去一个岗位为什么难以被 AI 接管？不是因为某个动作本身一定复杂，而是因为这个动作背后有大量隐性的业务上下文：为什么这样做、什么情况算异常、出了问题找谁、哪些历史坑不能再踩、哪些判断标准不能写错。如果这些上下文只在人脑里，AI 就只能做表层任务。

但当一个作用域的上下文、规则、历史和接口都被沉淀在文件夹里，人类就可以从这个作用域中逐步退出一部分重复工作。先让 Agent 观察，再让 Agent 汇总，再让 Agent 检查，再让 Agent 生成初稿，最后让 Agent 接管稳定流程。

所以这不是“用 AI 直接替换人”，而是：

```text
先外化上下文
再沉淀工作方式
再引入 Agent 运行
最后渐进替换执行器
```

人类被替换的前提，不是模型突然变聪明，而是组织先把原来藏在人脑里的上下文，变成 AI 可以运行的结构。

同样，这套结构也允许业务更快地更换人类执行者。

今天很多岗位之所以难交接，不是因为动作本身没人会做，而是因为上下文绑在具体人身上。一个人知道历史坑，知道谁靠谱，知道某个字段为什么不能改，知道某个需求当时为什么妥协。这个人一走，业务不是失去一个“执行动作的人”，而是失去一整块隐性上下文。

文件夹 Runtime 要解决的正是这个问题：让业务上下文不再私有化地挂在人身上，而是沉淀到作用域里。新人加入时，不是从翻群聊、问老人、猜历史开始，而是先进入这个业务文件夹，读取 README、历史材料、规则、风险和当前状态。

这样，人仍然重要，但不再因为“只有他知道上下文”而被组织锁死。业务可以更快换人，人也可以更自由流动。

换句话说，它同时支持两种替换：

```text
人类执行器 -> 新的人类执行器
人类执行器 -> AI Agent
```

前提都是一样的：业务上下文稳定存在，不跟着某个具体人一起流失。

## 一个具体例子：一个需求，一个文件夹

先不讲抽象概念，看一个非常具体的做法。

我们可以把一个需求相关的所有上下文，放在同一个 Git 仓库里的同一个文件夹下：

```text
单周迭代/
└── 2026/
    └── W15-04-06/
        └── 需求名/
            ├── PM/
            │   └── README.md
            ├── DS/
            │   └── README.md
            ├── RD/
            │   └── README.md
            └── QA/
                └── README.md
```

PM 的 `README.md` 是 PRD 和需求背景，旁边可以放用户反馈、竞品截图、SQL、会议转写。

DS 的 `README.md` 是设计说明，旁边可以放设计稿截图、交互补充、取舍记录。

RD 的 `README.md` 是技术方案，旁边可以放接口草稿、代码片段、风险判断。

QA 的 `README.md` 是测试用例和测试关注点，旁边可以放回归范围、验收截图、问题记录。

这件事第一步没有发生任何复杂变革，只是大家把工作的地方放到了同一个目录里。

它不要求 PM、设计、研发、QA 立刻变成全栈。每个人仍然在自己的角色文件夹里工作，只是同一个需求的上下文不再被学城、设计工具、IM 群、个人本地文件夹切碎。

对人来说，好处是不用到处找信息。

对 AI 来说，好处更直接：它不用每次重新拼上下文。只要读取这个需求文件夹，就能看到需求从哪里来，为什么这么设计，研发怎么判断，QA 准备怎么测。

外围系统也不需要立刻推翻。Ones、学城、迭代表、周报都可以继续存在，只是它们从“信息源”逐渐变成“视图”。真正的上下文源头，开始回到需求文件夹里。

## 问题一：为什么群聊不够？

企业协作今天最常见的载体是群聊。

一个需求来了，先拉需求群。研发介入后，又有开发群。测试阶段，有测试群。上线前，有上线群。出了问题，再拉故障群。一个需求可以自然裂变出很多群。

群聊的问题不是没有信息，而是信息很难稳定停留在可工作的状态里。

群聊是事件流。它适合通知、讨论、提醒、临时决策，但它不是稳定的状态空间。消息一多，历史就沉了；新人进来，很难知道哪些内容还有效；AI 进来，也不知道应该把哪几百条消息当成当前真实状态。

人类可以勉强处理这个问题，因为人会在脑子里重新建模：哪些消息是背景，哪些是结论，哪些已经过期，哪些需要跟进。

但这件事本质上是在消耗人的认知。每次新成员加入，每次 AI 介入，每次跨角色交接，都要重新做一遍上下文拼装。

所以群聊可以继续存在，但它不应该再承担“组织状态存储”的职责。

消息适合做状态变化通知。文件夹更适合保存当前状态。

## 问题二：协作的最小单元到底是什么？

如果不是群，那企业协作的最小单元是什么？

更合理的答案是：工作作用域。

一个需求是作用域。一个业务模块是作用域。一次线上事故是作用域。一个客户项目也是作用域。

作用域有几个特点：

- 它有参与者。
- 它有材料。
- 它有历史。
- 它有当前状态。
- 它有风险和待办。
- 它有生命周期。
- 它有对外输出。

群聊只是作用域的一种临时沟通界面。真正稳定存在的，是这个局部工作空间本身。

比如“退款”这个模块，可能存在很多年。人会换，群会换，IM 工具也会换，但“退款”这个业务作用域一直存在。它有自己的历史决策、接口约束、事故记录、负责人、上下游依赖。

这类作用域，如果只存在于人的脑子、群聊历史和散落文档里，就很难被继承，也很难被 Agent 长期维护。

所以组织真正应该沉淀的，不只是“谁做了什么”，而是这个业务作用域的上下文：它为什么存在，如何判断好坏，过去踩过什么坑，当前处在什么状态，未来变化会影响谁。

人是上下文的使用者和维护者，Agent 也是。执行器可以换，但上下文不能每次重建。

而文件夹天然就是作用域的物理形态。它能包含文件，能嵌套子目录，能做权限边界，能被索引，能被版本化，也能被 AI 读取。

所以，AI 时代的第一步不是“多建几个知识库”，而是重新识别组织里的工作作用域，并把它们落成一个个文件夹。

这相当于把组织架构从“人和岗位的关系”，扩展成“业务上下文的关系”。

过去组织问的是：这个事情归谁管？

现在还要继续问：这个事情的上下文在哪里？谁维护它？哪个 Agent 可以读它？它对外暴露什么接口？它的状态如何向上汇总，又如何被下游使用？

## 问题三：多个文件夹之间怎么协作？

如果一个文件夹是一个作用域，那么多个作用域就会自然形成文件树。

```text
公司/
├── 直播业务/
│   ├── 单周迭代/
│   ├── 收银台/
│   └── 退款/
└── 增长业务/
    ├── 活动/
    └── 投放/
```

树状结构的好处不是“看起来整齐”，而是它天然解决了三个问题。

第一，**上下文隔离**。不是每个人、每个 Agent 都需要看到全局细节。一个需求目录只需要维护自己的局部状态。

第二，**权限继承**。父目录可以定义大的权限和规则，子目录在这个范围内工作。

第三，**状态聚合**。父目录不需要读取所有子目录的内部过程，只需要读取每个子目录对外暴露的状态接口。

这个状态接口就是 README。

README 不应该只是“说明文档”，它应该成为一个目录对外暴露的抽象：

```text
这个目录当前在做什么？
现在进展到哪一步？
有什么风险？
依赖谁？
对外提供什么能力？
谁负责维护？
外部如果要接入，应该先看哪里？
```

这样一来，目录之间的通信就不再依赖“把所有人拉到一个群里看全量聊天记录”。

父目录读取子目录 README，做向上汇总。

子目录读取父目录 README，继承目标和约束。

兄弟目录之间不互相翻内部材料，只订阅对方暴露出来的状态、接口和风险。

这很像组织里的模块化。每个目录内部可以很复杂，但对外要有清晰接口。

## 问题四：AI 进来之后，文件夹为什么会变成 Runtime？

如果文件夹里只有人，它主要是一个信息组织方式。

但当 Agent 进来之后，文件夹就不只是“存储”了，它开始有运行时特征。

Agent 可以做几类事情：

- 定时读取文件夹，汇总最新状态。
- 发现 README 和内部材料不一致，提醒维护者更新。
- 读取子目录状态，生成父目录日报或周报。
- 根据目录里的 checklist 做例行检查。
- 根据历史记录判断风险是否重复出现。
- 在需求变更后，提醒 RD 和 QA 关注影响范围。

现在工具侧已经开始出现这种能力。OpenAI Codex 的 Automations 支持后台定时运行任务；Claude Code 的 common workflows 里也有 scheduled tasks 和 `/loop`，可以重复执行 prompt、轮询状态、检查 PR 或做提醒。

这说明 Agent 正在从“一问一答的工具”，变成“能定时醒来、读取上下文、执行动作、写回结果的运行时进程”。

一旦 Agent 有了运行时能力，它最需要的就不是更多群消息，而是稳定的工作目录。

所以文件夹 Runtime 可以这样理解：

```text
文件夹提供状态。
README 提供接口。
原始材料提供事实。
规则文档提供判断方式。
自动化任务提供触发机制。
Agent 提供运行时执行。
```

这也是为什么“skill 属于目录，不属于人”会变得重要。

一个人在某个目录里长期工作，会形成稳定的判断方式：怎么评估风险，怎么验收，什么时候需要找谁，哪些坑反复出现。过去这些经验往往存在人脑里。人走了，经验就断了。

但如果这些判断方式被写成目录里的 Markdown、checklist、skill，Agent 就可以继续读取并执行。新人进入这个目录，也不是从零开始翻群聊，而是接手一个已经沉淀了工作方式的运行环境。

人类的角色会变成：

```text
进入一个作用域
→ 探索问题
→ 形成稳定工作模式
→ 写成规则和 skill
→ 让 Agent 接管可重复部分
→ 人继续去处理下一个不稳定问题
```

这不是简单的“AI 辅助人”。更准确地说，是人把自己在某个作用域里的工作模式，逐步编译成可以长期运行的信息结构。

## 问题五：信息到底应该怎么存？

如果 Agent 要长期运行，文件夹里不能只放最后结论。

传统知识库经常有一个问题：它存的不是事实，而是“某个人在某个时间点，用某个视角压缩过的一份结论”。

比如一次线上事故。

研发关心接口超时和调用链。

QA 关心为什么没有覆盖到这个边界。

产品关心需求变更为什么影响了流程。

老板关心影响范围、品牌风险和组织责任。

如果我们把所有视角强行合并成一份统一的《事故复盘总结》，它看起来完整，但其实已经丢掉了大量未来可能有用的信息。

AI 时代更好的做法，是把事实和视角分开：

```text
底层：原始事实
      会议录音、聊天记录、日志、代码、截图、监控数据、变更记录

上层：角色视角
      PM 关注什么，RD 关注什么，QA 关注什么，老板关注什么

结果：运行时生成
      需要 PM 纪要时生成 PM 纪要，需要 QA 复盘时生成 QA 复盘
```

换句话说，摘要不应该被当成长期真相。摘要更像缓存，是某个角色在某个场景下的运行时产物。

真正应该长期保存的，是高质量原始事实，以及能指导 Agent 如何阅读事实的角色规则。

这也是文件夹比传统在线文档更适合 AI 的原因：它不仅能放一份最终文档，还能把会议转写、截图、日志、表格、代码片段、设计稿和规则文档放在一起。它保存的不是一个静态结论，而是一块可以被重新解释的上下文。

当然，保留原始事实不等于乱丢材料。文件夹仍然需要基本结构、权限和入口文档。否则上下文会变成垃圾堆，Agent 也会被噪声拖垮。

所以更准确的原则是：

> 保留原始事实，但用目录结构和 README 管住入口；延迟生成结论，但用角色规则约束视角。

## 问题六：怎么渐进式替换？

这套东西不能一上来就做成“全公司 AI 组织改造”。

更现实的路径，是先替换协作中心，再逐渐引入运行能力。

### 第一步：把上下文放一起

从一个需求开始。

建一个共享文件夹，让 PM、设计、研发、QA 都把和这个需求有关的材料放进去。先不用追求完美结构，只要保证一个原则：同一个需求的上下文不要散。

这一步解决的是“AI 和新人从哪里开始读”的问题。

### 第二步：建立固定入口

每个角色目录都要有 `README.md`。

README 不需要很长，但要回答当前状态、关键结论、风险、负责人、下一步。过程材料可以自由放，但入口要稳定。

这一步解决的是“外部怎么理解这个目录”的问题。

### 第三步：让 AI 做定时观察

先不要让 Agent 直接接管复杂工作。可以从观察、汇总、提醒开始。

比如每周自动读取本周所有需求目录，生成：

- 哪些需求状态不清楚。
- 哪些需求缺少设计或测试信息。
- 哪些 README 很久没更新。
- 哪些风险被多个角色重复提到。

这一步解决的是“目录会不会自己变旧”的问题。

### 第四步：把稳定工作沉淀成 skill

当某个目录里的工作方式稳定下来，就把它写成规则。

比如一个 QA 目录可以沉淀：

- 每次需求测试前必须检查哪些材料。
- 哪类变更必须补充回归。
- 哪些历史问题容易复发。
- 什么情况下要提醒 RD 或 PM。

这些规则一旦写进目录，Agent 就可以反复执行。

这一步解决的是“组织经验如何不靠人脑续命”的问题。

### 第五步：让 Agent 驻守局部作用域

最后，某些目录会逐渐变成半自动运行的局部系统。

人类仍然负责判断新问题、处理例外、做关键决策；Agent 负责例行观察、汇总、提醒、检查、同步、生成初稿。

这不是让 AI 一次性替代组织，而是让组织里的一个个局部作用域，逐步具备自我维护能力。

换句话说，文件夹 Runtime 给“渐进式替换人类执行器”提供了一条现实路径。

不是先问“这个岗位能不能被 AI 替代”，而是先问：

- 这个岗位所在的作用域是否清楚？
- 上下文是否稳定沉淀？
- README 是否能表达当前状态？
- 判断规则是否可以写出来？
- 例行工作是否可以被定时触发？
- 异常边界是否可以被识别并上抛给人？

这些问题回答得越清楚，人类就越容易从重复执行中退出，转向处理新的、不稳定的、尚未被结构化的问题。

## 和 Context, not Control 的关系

这套思路和张一鸣在 2017 年源码资本码荟上讲的 **Context, not Control** 是同一个方向。

张一鸣当时讲的是公司变大后的管理问题：早期 CEO 可以像“超级计算机”一样做很多判断和控制，但公司变大、业务变复杂之后，单点控制会变慢，也会因为信息不充分而失真。更好的方式，是让更多人拿到决策所需的上下文，基于充分信息自己判断。

他对 Context 的定义很关键：Context 不是一句背景介绍，而是“决策所需要的信息集合”，包括原理、市场环境、行业格局、优先级、业务目标、业务数据、财务数据等。Control 则是委员会、指令、分解、汇总、流程、审批。

从这个角度看，文件夹 Runtime 可以理解为 **Context, not Control 的 AI 时代基础设施化版本**。

2017 年讲的是管理哲学：

```text
不要把所有决策收回到上级。
要让一线的人掌握充分上下文，自己做判断。
```

现在我们要解决的是更底层的问题：

```text
这些上下文到底放在哪里？
怎么保持更新？
怎么让新人和 Agent 都能读懂？
怎么让上下文不只服务人，也能驱动自动化任务？
```

过去的 Context 更多依赖人、会议、群聊、OKR 系统、内部文档和信息透明文化。它能让人更好决策，但仍然有一个问题：上下文容易散。

文件夹 Runtime 则试图把 Context 落成一个稳定的信息结构：

- 一个作用域一个文件夹。
- README 暴露当前状态。
- 原始材料保留事实。
- 角色规则定义视角。
- 自动化任务定时检查。
- Agent 驻守并写回结果。

这样，Context 不再只是“管理者提供给员工的信息”，而变成一个可以被人和 Agent 共同读取、维护、运行的工作空间。

这也解释了为什么这套设计不是 Control。

它不是给每个人增加流程，不是让所有动作都审批，也不是把组织拆成互不协作的小 BU。相反，它是在减少实时控制：通过把上下文建好，让局部作用域内的人和 Agent 能更自主地判断。

这里有一个关键变化：组织不再假设某个固定的人长期掌握所有背景，也不再假设某个固定工具长期承担所有执行。人可以换，执行器可以换，但业务上下文要稳定存在。只要上下文稳定，新人可以接手，新的 Agent 可以接手，新的自动化也可以接手。

但它也不是完全无规则。文件夹命名、README 结构、权限边界、关键节点检查、合规底线，这些仍然需要少量 Control。真正的原则不是“不要控制”，而是：

> 用最少的控制，维护最高质量的上下文。

张一鸣当年说，建设 Context 这个基础工程并不容易，需要大量沟通、管理和产品技术工作。文件夹 Runtime 正好可以接住这句话：它把“建设 Context”从一种管理要求，变成一种可以被产品化、工具化、自动化的信息系统设计。

## 这会改变什么？

第一，沟通的对象会变。

过去我们主要把信息发给人。以后很多时候，我们是在把上下文放进一个作用域，让人和 Agent 都能读取。

第二，交接的方式会变。

过去交接靠会议、口述、在线文档和群聊历史。以后更理想的交接，是把一个维护良好的文件夹交给下一个人或下一个 Agent。

第三，组织记忆的位置会变。

过去组织记忆主要存在于老人脑子里、散落文档里和不可追溯的聊天记录里。以后它应该更多存在于目录结构、README、原始事实、规则文档和驻守 Agent 中。

第四，人的价值会变。

未来更有价值的人，不只是“能把事做完”的人，而是能把一个作用域做清楚、沉淀清楚、接口定义清楚，并逐步让 Agent 接手可重复工作的那类人。

这类人做的不是简单文档整理，而是在做一种新的组织工程：把临时经验编译成长期可运行的信息结构。

也因此，未来组织里的岗位不会突然消失，而会先发生“执行器替换”：同一个业务上下文还在，同一套目标、规则、风险和接口还在，只是其中一部分原本由人执行的动作，逐渐变成由 Agent 执行。

第五，组织韧性会变强。

当上下文存在于文件夹里，而不是绑定在某个人身上，换人、轮岗、扩团队、引入外包、引入 Agent 的成本都会下降。组织不再那么依赖“某个具体的人一直在场”，而是依赖“这个业务作用域的上下文是否被持续维护”。

## 最后

文件夹不是资料柜。

在 AI 时代，它会变成组织局部状态的运行单元。

群聊仍然有价值，它负责触发、讨论和通知。但真正长期存在的，不应该是聊天窗口，而应该是一个个可读取、可追溯、可运行的工作作用域。

一个需求一个文件夹，是最小的开始。

README 是这个文件夹的接口。

原始材料是事实底座。

规则文档是角色视角。

定时任务和 Agent 是运行时。

当这些东西连起来，组织就不再只是人和人之间的消息网络，而会逐渐变成一棵持续运行的文件树。

这可能是 AI 时代企业协作最底层的一次变化：从“传递消息”，走向“维护上下文”；从“写文档”，走向“构建可运行的组织状态”。
