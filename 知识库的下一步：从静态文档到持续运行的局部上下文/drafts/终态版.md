# 从群聊到文件夹：AI 时代的组织信息设计

![文件夹 Runtime](./assets/folder-runtime-hero.png)

## 先说最终态

AI 时代，组织会越来越像一棵持续运行的文件树。

这棵树上的每个文件夹，都是一个业务作用域：一个需求、一个模块、一次事故、一个客户项目，最终都应该被建模成一个文件夹。

文件夹里不只是放文档，而是沉淀这个局部业务作用域的完整上下文：目标、材料、历史、状态、风险、规则、负责人、上下游关系，以及它过去如何被理解和处理。

人可以进入这个文件夹工作。

Agent 也可以进入这个文件夹工作。

人可以换，Agent 可以换，工具可以换，但业务上下文不应该每次重建。

这是这套设计最核心的结论：

> 组织真正稳定的资产，不是某个具体的人，也不是某个具体执行器，而是一个个业务作用域的上下文。

传统组织架构图回答的是：谁负责什么。

AI 时代还需要另一层结构回答：

```text
这个业务的上下文在哪里？
谁维护它？
谁可以读取它？
它如何对外暴露状态？
Agent 如何在其中工作？
它如何支持换人、交接和自动化？
```

所以这不是简单地整理文档，也不是重画组织架构图。

它是在重新设计组织架构背后的上下文层。

## 先从低迁移成本的形态开始

以一个需求为例。

这不一定是最终形态，只是最容易迁移、最容易理解的第一层做法。

一个需求不应该只存在于学城文档、设计稿链接、研发方案、测试用例、群聊记录和几个人的脑子里。

它应该首先是一个文件夹：

```text
需求名/
├── PM/
│   ├── README.md
│   └── facts/
├── DS/
│   ├── README.md
│   └── facts/
├── RD/
│   ├── README.md
│   └── facts/
├── QA/
│   ├── README.md
│   └── facts/
└── facts/
    └── 跨角色会议、需求级群聊、全局决策记录
```

PM 在 `PM/` 里放 PRD、调研、数据分析和产品判断；PM 自己产生的调研截图、访谈转录、分析表格，可以放在 `PM/facts/`。

设计在 `DS/` 里放设计说明、交互补充、取舍记录；设计过程材料可以放在 `DS/facts/`。

研发在 `RD/` 里放技术方案、接口草稿、风险判断；研发过程中的日志、实验、代码分析可以放在 `RD/facts/`。

QA 在 `QA/` 里放测试用例、回归范围、验收记录；测试过程证据可以放在 `QA/facts/`。

需求根目录下的 `facts/` 只放跨角色、需求级的原始事实。

比如 PM 和设计之间的小会，如果主要服务于产品和设计取舍，可以放在 `PM/facts/` 或 `DS/facts/`，再在对应 README 里写出结论；如果它影响整个需求范围、排期、研发实现或测试策略，就应该放到需求根目录的 `facts/`。

也就是说，事实材料跟着作用域走：角色内部事实放角色目录，跨角色事实放需求根目录。未来无论谁有问题，都可以从对应作用域回到原始事实重新提问。

每个角色目录都有一个固定入口：`README.md`。

README 不需要成为长篇大论，它只是这个目录对外暴露的当前状态：现在做什么，关键结论是什么，有什么风险，谁负责，下一步是什么。

这样，一个需求的上下文就不再散落。

人进来，可以从文件夹开始读。

AI 进来，也可以从文件夹开始读。

需求负责人换了，新人进入这个文件夹，就能看到历史、当前状态和关键判断。

某些工作稳定后，Agent 可以进入这个文件夹工作：检查 README 是否过期，汇总风险，发现 PM、设计、研发、QA 是否状态不一致，把结果写回文件夹或推给负责人。

到这个阶段，这个文件夹就不只是“资料存放地”。

它开始变成一个业务上下文的运行空间。

也就是文件夹 Runtime。

## 群聊和文件夹是什么关系？

这套设计不是要废弃群聊。

更准确地说，群聊应该绑定到文件夹上。

一个群，本质上也是一个局部作用域。一个需求群、开发群、测试群、上线群、故障群，背后都有一块局部信息和一组参与者。

只是过去这个局部作用域没有稳定的状态本体，所以大家只能把群聊本身当成工作空间。

一个需求推进过程中，会自然裂变出很多群：需求群、开发群、测试群、上线群、故障群。群里当然有信息，但它是一条不断往下流的事件流。

消息会沉没。

结论会过期。

新人不知道哪条还有效。

AI 也不知道应该把哪几百条消息当成当前真实状态。

人类之所以能靠群聊工作，是因为人会在脑子里重新整理上下文：哪些是背景，哪些是结论，哪些已经作废，哪些需要跟进。

但这件事其实一直在消耗人的认知。

每次换人、交接、跨角色协作、让 AI 介入，都要重新做一遍上下文拼装。

所以，不是从“群聊”转向“没有群聊”。

而是从“群聊本身就是协作中心”，转向“文件夹是协作中心，群聊是绑定在文件夹上的沟通界面”。

一个新型业务作用域可以这样理解：

```text
文件夹：保存上下文、状态、材料、规则、历史
群聊：承载讨论、通知、临时对齐
人类：在文件夹和群聊之间工作
Agent：在文件夹和群聊之间工作
```

人和 Agent 都可以在这个文件夹工作，也都可以出现在绑定的群里。

群里发生的关键讨论，最终应该回写到文件夹；文件夹里的状态变化，也可以通过群通知出来。

真正长期存在的，不应该是聊天窗口，而应该是业务作用域的文件夹。

## 从聊天框、文档到文件夹

过去十几年，组织信息的载体其实经历过几次升级。

第一阶段是聊天框。

聊天框解决的是速度问题。它让人能快速对齐、快速通知、快速响应。但聊天框的信息会下沉，结构弱，时间一长就很难承载复杂上下文。

第二阶段是在线文档。

飞书文档、Google Docs、Notion 这类工具，相比聊天框大幅提高了信息带宽。它们让组织可以把思考、方案、流程、复盘写下来，也让更多人能异步阅读和评论。

这其实很符合张一鸣过去讲的 Context, not Control：组织不应该只靠控制和审批运转，而应该让更多人拿到足够上下文，基于上下文自己判断。

但在线文档也有代价。

它要求人有比较强的写作能力、结构化能力和文档维护习惯。一个组织想靠文档跑起来，背后其实需要一批受过良好训练、愿意长期写文档的人。

而且，文档仍然偏向“文字化后的结论”。很多真实上下文并不天然长成文档：截图、录音、日志、表格、代码片段、设计稿、会议转写、临时实验、历史版本，它们常常被文档引用、压缩、摘录，但很少完整待在同一个上下文里。

AI 时代之后，情况变了。

Agent 天生擅长读写文件夹。它可以读取 Markdown，也可以读取图片、表格、代码、日志、转写和各种原始材料。它不一定需要所有人先把信息整理成一篇漂亮文档，再交给它理解。

这意味着，组织信息的中心可以继续往前走一步：

```text
聊天框：解决即时沟通
在线文档：提升文字化信息带宽
文件夹：承载完整多媒体上下文
```

飞书文档不是没有价值，它仍然适合展示、协同写作、评论和分发。

但在 AI 原生的工作方式里，文档更像是文件夹上下文的一种视图，而不是唯一的信息源。

每个人不一定都要围绕同一篇文档阅读同一个结论。

更可能的方式是：大家围绕同一个文件夹工作，每个人和自己的 Agent 根据自己的问题去问这个文件夹。PM 问产品判断，RD 问技术风险，QA 问测试范围，负责人问整体进度。

同一份上下文，可以被不同角色实时投影成不同答案。

这就是文件夹比单篇文档更适合 AI 时代的原因。

## 为什么说文件夹是业务作用域？

一个业务作用域不是一个群，也不是一篇文档。

它是一块能局部闭环的工作空间。

比如一个需求，它有目标、背景、参与者、设计、方案、测试、风险、历史决策和上线结果。

比如一个模块，它有接口、上下游、事故记录、负责人、历史坑和未来规划。

比如一次事故，它有日志、时间线、影响范围、处理过程、复盘结论和后续改进。

这些东西共同构成了这个作用域的上下文。

过去这些上下文经常绑在人身上。

一个老同事知道为什么不能改某个字段，知道某个需求为什么妥协，知道某个合作方为什么难推进。这个人一走，组织丢掉的不只是一个执行者，而是一整块隐性上下文。

文件夹结构要解决的，就是让业务上下文不再私有化地挂在某个人身上。

业务上下文沉淀在文件夹里。

人只是它的维护者和执行者。

Agent 也是它的维护者和执行者。

这样，执行器可以换，但业务上下文还在。

## 接口人和接口 Agent

如果文件夹里什么都能放，它很快也会变成垃圾堆。

所以每个文件夹都需要对外接口。

最简单的接口是 README。

README 的作用不是写一篇漂亮文档，而是回答外部最需要知道的几个问题：

```text
这个目录现在是什么状态？
关键结论是什么？
有什么风险？
谁负责？
下一步是什么？
外部如果要了解这里，应该先看哪里？
```

目录内部可以放大量过程材料，但对外要有一个清晰边界。

父目录可以读取子目录 README，汇总成周报和风险列表。

兄弟目录可以读取彼此 README，理解依赖关系。

Agent 可以定时读取 README，发现它和内部材料是否不一致。

但 README 只是最朴素的接口形态。

更完整的结构，应该是：

```text
文件夹内部：完整上下文、原始材料、历史、规则
对外接口：README + 接口人 / 接口 Agent
```

这个接口人可以是人类，也可以是专门的 Agent。

它的职责不是把整个文件夹暴露出去，也不是提前写出所有中间总结。

它更像一个会读目录的接口：外部带着自己的问题来问，它根据这个问题，把目录里的上下文现场投影成对方需要的答案。

比如同一个需求文件夹：

- PM 问：这个需求的核心取舍是什么？
- RD 问：这里有哪些技术风险？
- QA 问：哪些边界必须回归？
- 负责人问：当前最大的阻塞是什么？
- 上游目录问：这个需求会不会影响我的排期？

这些问题不应该都依赖同一份提前写死的总结。

因为不同角色想看的不是同一份“标准答案”，而是同一个目录在自己问题下的不同投影。

这和现在看代码很像：你不会先要求别人把整个代码仓库总结成一篇万能文档，而是直接问 AI 一个具体问题。AI 会临时爬目录、读相关文件、理解调用关系，然后给出贴合你当前问题的解释。

业务文件夹也应该这样。

不是提前总结一切，而是保留完整上下文，让人或 Agent 带着问题来问，再由接口人或接口 Agent 现场计算出答案。

这样也更自然地解决权限问题。

外部不一定能读完整文件夹，也不一定需要读完整文件夹。它可以向这个文件夹的接口人提问。接口人根据权限和视角，返回必要信息。

所以文件夹之间的通信，不是互相复制全部上下文。

而是通过提问和回答来通信：父目录问当前风险，兄弟目录问接口变化，外部角色问自己这个视角下需要注意什么。

这就像组织里的“接口人”，只是在 AI 时代，这个接口人可以逐渐变成接口 Agent。

README 仍然重要，但它不必承担所有表达。

它更像一个默认入口；真正灵活的接口，是能基于完整上下文即时回答问题的人或 Agent。

这里还要注意一点：目录不是固定分成几种“类型”，而是对不同的人和 Agent 有不同的可见性。

一个人或 Agent 在某个目录里工作时，这个目录就是它的当前作用域。它通常可以读写这个目录里的完整上下文。

但它不应该天然拥有外部所有目录的完整读写权限。外部目录如果和当前工作有关，应该以某种形式被引用进来。

这里可以有一个很简单的规则：

**如果一个目录声明了接口人，外部默认就不直接读取它的内部细节，而是先找接口人。**

更进一步，外部目录可能都不应该被直接展开。

本目录真正需要维护的，是对外部依赖的索引：依赖哪个外部作用域，应该找哪个接口人，能问哪些问题。

比如一个需求依赖财务、法务、平台能力或其他业务模块，本目录里不需要直接挂进那些外部目录的完整材料。它只需要记录：

```text
依赖：财务结算规则
接口人：财务接口 Agent
可问问题：结算口径、风险、审批要求
```

当本目录的人或 Agent 需要这些信息时，就去问对应的外部接口人。

换句话说，不同外部目录对当前 Agent 来说，有不同的展开方式：

```text
本目录：可以读写完整上下文
外部目录：默认通过本目录记录的接口人 / 接口 Agent 提问
```

外部目录不是空壳。它背后仍然是一个真实运转的业务作用域。只是对当前目录来说，它不表现为一堆可随意读取的文件，而表现为一个可查询的接口。

接口人其实就是这个目录的导诊台。

外部不知道该看什么，不知道该问谁，不知道内部哪些细节可见，就先问接口人。接口人负责路由、解释、压缩和权限判断：哪些信息能说，哪些不能说；不同角色问同一个问题时，应该给出什么粒度的回答；什么时候可以直接回答，什么时候必须上抛给人。

如果接口人是 Agent，它就是这个复杂目录对外的默认入口。它不一定替代内部所有人，但它可以先承担导诊、路由、状态解释和即时投影的工作。

这部分未必一开始就需要新工具。早期可以先靠权限、README、固定联系人、Agent 约定来模拟。等这种模式稳定后，再考虑是否需要更明确的目录引用和接口机制。

## 为什么 AI 进来后，文件夹会变成 Runtime？

如果只有人使用文件夹，它主要是信息组织方式。

但 Agent 具备循环执行、工具调用、定时醒来和写回能力后，文件夹就开始像一个运行环境。

Agent 在文件夹里工作，自然就会包含一些运行时动作：

- 看 README 有没有过期。
- 看 PM、设计、研发、QA 的状态是否一致。
- 看风险是否被反复提到。
- 看哪些待办没人处理。
- 根据规则生成提醒。
- 把结果写回文件夹。

OpenAI Codex 的 Automations、Claude Code 的 scheduled tasks 和 `/loop` 都说明了这个趋势：Agent 不再只是被临时问一句的工具，它开始可以周期性地读取上下文、执行动作、写回结果。

这时，文件夹就不只是存储。

它变成了 Agent 可以长期驻守的业务运行空间。

所以文件夹 Runtime 可以这样理解：

```text
文件夹提供上下文
README / 接口 Agent 提供对外接口
规则文档提供判断方式
Agent 在其中执行、观察、更新和沉淀
```

## 为什么建设上下文，而不是建设工具？

这个问题很关键。

我们当然也可以去做工具：做一个新的协作系统，做一个新的 AI 工作台，做一个新的知识库产品。

但工具这条路风险很大。

世界上最聪明、资源最多的一批人，正在模型厂商和工具厂商里建设这些东西。OpenAI、Anthropic、Cursor、GitHub、Notion、Linear、Slack 都在往 AI 工作空间、Agent、自动化、长上下文、工具调用的方向推进。

如果我们把主要精力放在自研工具上，很容易出现一个问题：还没做完，外部工具和模型就已经往前走了一大步。我们辛苦建设的能力，可能直接被模型厂商或工具厂商的新版本覆盖。

但建设上下文不一样。

业务上下文是我们自己的资产。模型厂商不会替我们整理直播业务的历史决策，不会知道某个需求为什么妥协，不会知道某个字段为什么不能改，也不会自动理解我们团队真实的协作边界。

我们最应该投入的，不是和全世界最强的工具团队比赛造工具，而是把自己的业务上下文沉淀好。

这样做有一个很大的好处：任何工具进步，我们都能吃到。

今天的 Agent 能做简单汇总，它就读这些文件夹做简单汇总。

明天的 Agent 能做复杂推理，它就读这些文件夹发现更深的风险和方案。

后天的模型上下文更长、推理更强、工具调用更稳，它仍然可以进入同一批业务文件夹，从旧事实里发现过去看不出来的新洞察。

如果我们只存一份提前压缩过的结论，未来模型再聪明，也很难从有损摘要里推回真实事实。

但如果我们沉淀的是完整上下文，模型每进步一次，这些上下文的价值就会被重新释放一次。

所以，工具会变，模型会变，Agent 会变。

我们真正要建设的是能长期复利的上下文资产。

## 它为什么允许渐进式替换？

AI 替换人的前提，不是模型突然变聪明。

更现实的前提是：组织先把原来藏在人脑里的上下文外化出来。

一个岗位之所以难以被 AI 接管，往往不是因为每个动作都很难，而是因为动作背后有大量隐性判断：什么时候算异常，哪些历史坑不能再踩，出了问题找谁，哪些风险必须提前说。

如果这些东西都沉淀在文件夹里，替换就可以渐进发生：

```text
先让 Agent 观察
再让 Agent 汇总
再让 Agent 检查
再让 Agent 生成初稿
最后让 Agent 接管稳定流程
```

同样的逻辑也适用于换人。

当业务上下文在文件夹里，而不是绑在某个具体人身上，一个新人就能更快接手。业务不再因为某个人离开而上下文断裂。

所以这套结构支持两种替换：

```text
人类执行器 -> 新的人类执行器
人类执行器 -> AI Agent
```

真正稳定的不是执行器，而是业务上下文。

## 这和 Context, not Control 有什么关系？

这套思路和张一鸣讲的 **Context, not Control** 是同一个方向。

Context, not Control 的核心是：不要靠层层控制让组织运转，而是让更多人拿到充分上下文，基于上下文自己判断。

从这个角度看，飞书文档这类工具，可以理解为上一阶段建设 Context 的重要载体。它比聊天框更稳定，比口头同步更可追溯，也更适合组织沉淀文字化的思考。

但在 AI 时代，我们还要继续往下问一步：

这些上下文到底放在哪里？

怎么保持更新？

怎么让新人能读懂？

怎么让 Agent 也能读懂？

怎么让上下文不只是帮助人决策，还能驱动自动化任务？

文件夹 Runtime 可以理解为 Context, not Control 的基础设施版本。

它不是否定文档，而是把 Context 从“写出来的一篇文档”，继续扩展成“一个可被人和 Agent 共同读取的完整上下文空间”。

它也不是增加控制，而是减少对具体人的依赖。通过把上下文建设好，让局部作用域里的人和 Agent 都能更自主地工作。

真正的原则不是不要控制，而是：

> 用最少的控制，维护最高质量的上下文。

## 怎么一步步做到？

不用一步到位。

第一步，从一个需求开始。

建一个需求文件夹，让 PM、设计、研发、QA 把相关材料放进去。先不要追求完美结构，只要保证同一个需求的上下文不要散。

第二步，固定 README。

每个角色目录都要有 README，用来说明当前状态、关键结论、风险和下一步。过程材料可以自由放，但入口要稳定。

第三步，让 AI 做观察者。

先不要让 Agent 直接接管复杂工作。可以让它每周读一遍目录，提醒哪些 README 过期、哪些需求缺少测试信息、哪些风险反复出现。

第四步，把稳定工作写成规则。

比如测试前必须检查什么，需求变更后要提醒谁，哪些历史问题容易复发。规则写出来之后，Agent 才能反复执行。

第五步，让 Agent 驻守部分目录。

当一个作用域里的上下文足够稳定、规则足够清楚、异常边界足够明确，人类就可以逐渐退出重复执行，去处理新的、不稳定的、还没有被结构化的问题。

## 最后

表面上看，这只是“一个需求一个文件夹”。

往深了看，它是在重新设计组织上下文。

过去组织依赖人记住上下文、传递上下文、解释上下文。

AI 时代，组织需要把上下文变成一种稳定资产：能被人读取，也能被 Agent 读取；能被新人继承，也能被自动化任务运行。

群聊仍然重要，但它更适合做通知和讨论。

真正长期存在的，应该是一个个业务作用域的文件夹。

人可以换。

执行器可以换。

但业务上下文要留下来。

当上下文被稳定沉淀，组织才可能真正进入 AI 原生的协作方式。
