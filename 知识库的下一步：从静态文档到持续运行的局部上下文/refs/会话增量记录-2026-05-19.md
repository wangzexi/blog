# 会话增量记录：文件夹协作文章

这份记录来自 2026-05-19 这轮文章整理过程，用来保留原始资料里没有被充分显性化、但对后续写作很关键的信息。

## 这轮对话补出来的核心信息

### 1. 主线不是“AI 做低风险辅助”

文章的核心不是证明 AI 可以先做一些低风险辅助工作。

更准确的结构是：

```text
业务文件夹 = 局部业务上下文 + 人的工作面 + AI 的工作面 + 定期运行的任务
```

人围绕这个文件夹工作。

AI 也围绕这个文件夹工作。

当某些劳动模式稳定后，人可以把一部分劳动派给文件夹里的 AI 助手，由它定期读取、执行、写回结果。

所以“查漏补缺、汇总、生成初稿”只是例子，不应该成为文章主叙事。主叙事应该是：文件夹变成人和 AI 共同工作的上下文运行面。

### 2. 过去为什么不用文件夹

文件夹一直存在，但过去没有成为组织协作中心，不是因为文件夹不对，而是因为人处理不了高带宽原始信息。

一个文件夹里如果放满会议录音、截图、群聊记录、Excel、SQL、日志、设计稿、临时草稿，对人来说很快会变成资料堆：

- 东西是全的，但不好找。
- 材料是原始的，但没人愿意一条条看。
- 每个角色只关心其中一部分，却要自己从噪声里筛。

所以过去更合理的工程妥协，是把原始材料压缩成文档。飞书文档、Google Docs、Notion 这类工具，本质上是在降低人的阅读成本。

### 3. 现在为什么可以用了

AI 改变的是“阅读层”。

现在人不一定要直接读完整文件夹，而是可以通过 AI 问文件夹：

```text
这个需求现在最大的风险是什么？
PRD 和最新群聊结论有没有冲突？
研发方案有没有漏掉设计约束？
QA 需要重点覆盖哪些边界？
```

过去是人直接读材料。

现在是人通过 AI 问材料。

这让文件夹重新变得可用：文件夹可以保存更高带宽的原始上下文，AI 负责按问题临时搜索、比对、组织和解释。

### 4. 原始材料的长期价值

提前写好的文档是一种有损压缩。

如果只存摘要，未来模型再强，也只能在旧摘要上推理，推不回真实事实。

如果保留原始上下文，模型每次升级后，都可以重新读取同一批事实，生成更好的视角、发现过去没发现的关联。

这也是“建设上下文，而不是建设工具”的关键理由：工具会变，模型会变，但业务事实和历史上下文是团队自己的长期资产。

### 5. 群聊和文档的位置

群聊不是废弃，而是绑定在文件夹上的沟通界面。

群聊负责讨论、通知、临时对齐；真正影响后续执行的结论，应该回到业务文件夹。

文档也不是废弃，而是文件夹上下文的一种视图。

过去是：

```text
原始材料 -> 人工压缩 -> 一篇文档 -> 大家读文档
```

现在更合理的是：

```text
原始材料 -> 留在文件夹 -> 人和 AI 带着问题读取 -> 生成当下需要的视图
```

### 6. 文件夹 Runtime

当 AI Agent 具备定时任务、循环执行、工具调用和写回能力之后，文件夹就不只是资料柜，而会变成 runtime。

这个 runtime 里至少包含：

```text
业务文件夹 = 上下文 + 当前状态 + 原始材料 + 规则 + 人 + AI Agent
```

Agent 可以每天或每周醒来，读取文件夹，检查状态、同步结论、生成风险和进度，并把结果写回文件夹或推送到绑定的群。

这里的重点不是“AI 临时帮一下”，而是“AI 驻守在一个稳定上下文里持续工作”。

### 7. 表达上的注意事项

不要把文章写成“替换人”。

更合适的表达是：

- 人把一部分稳定劳动派给文件夹里的 AI 助手。
- AI 在文件夹里定期执行稳定任务。
- 人继续负责目标、判断、沟通、取舍和责任。
- 文件夹让新人、老人、负责人、AI 都能面对同一个上下文。

也不要过度写“AI 低风险辅助”。这个说法可以用于降低误解，但不应该覆盖主线。

主线应该是：业务文件夹成为组织局部上下文的运行面，人和 AI 都在这个运行面上工作。
