## 第 10 轮 — ChatGPT

对，群与群之间就应该像**组件通信**，不能靠人肉转发。

可以这样建模：

```text
目录 = 组件
目录状态 = props/state
目录产出 = output
上级目录 = parent
下级目录 = child
同级目录 = sibling
Agent = controller/effect
```

核心问题是：**目录之间不能共享全部上下文，只能共享接口。**

否则会变成全局变量污染。

比较合理的机制是四类：

```text
1. 向上汇总
子目录 -> 父目录
输出摘要、风险、进度、阻塞、决策点

2. 向下分发
父目录 -> 子目录
分发目标、约束、背景、验收标准

3. 横向订阅
目录A -> 目录B
只订阅某些事件或状态字段，比如 API 变更、设计稿变更、排期变更

4. 显式引用
一个目录引用另一个目录的文件、结论、接口文档或状态快照
```

这就很像前端组件：

```text
父子通信：props / callback
兄弟通信：event bus / store / service
跨模块通信：API contract
```

放到企业 OS 里就是：

```text
父子目录：状态继承 + 汇总
兄弟目录：事件订阅 + 合约引用
跨项目目录：API / 文档 / Agent Bridge
```

关键不是“群之间互相聊天”，而是：

> 每个目录暴露一个稳定的状态接口。

比如一个目录可以暴露：

```text
/status
/risks
/decisions
/artifacts
```
