# 《文件树中心制》整理建议

这篇文章的核心不是“换一种文档存放方式”，而是：**AI 时代，组织需要把信息从消息流，迁移到可持续运行的状态空间。**

现在原文的问题不是观点不够，而是观点太多、跳跃太快。它同时想讲四件事：

1. 一个现实做法：一个需求一个文件夹。
2. 一个协作原则：文件夹是工作作用域，README 是接口。
3. 一个 AI 变化：Agent 需要状态空间，而不是聊天记录。
4. 一个组织理论：企业是持续运行的认知系统。

建议重写时只让读者每次跨一层台阶：先看到痛点，再看到例子，再意识到例子背后的结构，最后再接受“组织信息设计”的大结论。

## 推荐叙事顺序

### 1. 结论先行

开头不要先讲“新范式”，先讲一个更低阻力的结论：

> AI 时代，企业协作的核心载体会从“群聊”变成“文件夹”。
>
> 群聊适合人类临时沟通，但不适合 Agent 长期工作。Agent 需要的是一个能持续读取、更新、追溯、定时运行的状态空间。文件夹正好是这个空间。

然后给三句钉子：

- 文件夹不是资料容器，而是工作作用域。
- README 不是普通文档，而是作用域对外暴露的状态接口。
- Agent 不是被临时 @ 的工具，而是驻守在目录里的运行时协作者。

### 2. 先举一个具体例子

例子要尽量朴素，不要一上来就进入组织哲学。

推荐用 `mlive-context`：

```text
需求名/
├── PM/README.md
├── DS/README.md
├── RD/README.md
└── QA/README.md
```

这里要强调一句原始讨论里很强的话：

> 这件事第一步没有发生任何复杂变革，只是大家把工作的地方放到了同一个目录里。

这句话能让读者放下防御。然后再讲收益：

- PM 的调研、截图、会议转写、PRD 不再散落。
- DS 能读到需求背景。
- RD 能读到产品、设计、QA 的上下文。
- QA 能看到需求、方案、代码变化。
- AI 不用每次重新喂材料，可以直接读整个需求文件夹。

### 3. 用问题层层递进

建议把第三部分改成“问题推导”，而不是现在的“五层抽象”。读者会更轻松。

#### 问题一：为什么群聊不够？

因为群聊是事件流，不是状态空间。

群聊的问题不是信息少，而是信息不能稳定停留在可工作的形态里：会沉没、不可继承、不可版本化、边界混乱、新人难以进入。

#### 问题二：那协作的最小单元到底是什么？

不是群，而是工作作用域。

一个需求、一个模块、一个事故、一个客户、一个项目，都是一个可以局部闭环的工作作用域。它有参与者、材料、状态、历史、风险、输出和生命周期。

文件夹天然就是作用域的物理形态。

#### 问题三：多个作用域怎么组织？

用文件树。

树解决三件事：上下文隔离、权限继承、状态聚合。每个目录只维护自己的局部状态，并通过 README 对外暴露摘要、风险、接口、负责人。

#### 问题四：AI 进来之后，文件夹为什么从“存储”变成“runtime”？

因为 Agent 不是只读文档，它会周期性读取状态、执行任务、写回结果、触发下一步。

现在工具侧已经在往这个方向走：Codex 有 automations，可以在后台按计划运行任务；Claude Code 有 `/loop` 和 scheduled tasks，可以反复执行 prompt、轮询构建、检查 PR 或做提醒。也就是说，Agent 开始具备“定时醒来、读取上下文、执行动作、写回状态”的运行时特征。

所以文件夹不再只是资料柜，而变成：

```text
文件夹 = 状态空间 + 规则 + 入口文档 + 自动化任务 + Agent
```

这就是“文件夹 runtime”。

#### 问题五：信息到底应该怎么存？

这里再引入“事实与视角解耦”，但要压短，让它服务主线。

不要把所有原始事实提前压成一份统一结论。正确方式是：

```text
底层：原始事实
上层：角色视角
结果：运行时生成
```

同一场会议，PM、RD、QA、老板关心的东西不同。AI 时代更好的做法，是保留原始材料，让不同角色在需要时按自己的视角生成纪要、风险、方案和复盘。

#### 问题六：组织如何渐进式替换？

不要一上来讲“AI 组织”。建议分四步：

1. **先把上下文放一起。** 一个需求一个文件夹。
2. **再建立固定入口。** 每个目录有 README，作为状态接口。
3. **再让 AI 做定时观察。** 每天/每周自动汇总风险、变更、待办。
4. **最后让 Agent 驻守目录。** 当某个目录里的工作模式足够稳定，就把 checklist、判断规则、操作流程沉淀为 skill，让 Agent 接管可重复部分。

这条路径很重要，因为它回答老板最关心的问题：不是推倒重来，而是渐进替换。

## 建议删减

原文里这些地方建议压缩：

- “操作系统、分布式系统、React 组件树、神经网络层级”这组类比可以删掉或后置到脚注。它们对工程师有吸引力，但会打断主线。
- “企业是持续运行的认知系统”不要太早出现。放在最后收束，否则读者还没接受文件夹，就被推到哲学层。
- “事实与视角解耦”值得保留，但从一大节压成一个短节。它是主论点的支撑，不要让它抢主角。
- “组织结构即文件树”可以改成更柔和的说法：组织的信息结构，应该越来越像一棵持续运行的文件树。

## 可直接替换的开头草稿

# 从群聊到文件夹：AI 时代的组织信息设计

AI 时代，企业协作的中心会从“群聊”逐渐转向“文件夹”。

不是因为文件夹更先进，而是因为 Agent 的工作方式变了。

人类可以靠群聊工作。我们看到消息、理解上下文、记住历史、判断轻重缓急，再把散落的信息重新组织成行动。但 Agent 不适合长期生活在消息流里。消息会沉没，状态会断裂，历史不可继承，新的 Agent 每次加入都要重新喂上下文。

Agent 更需要一个稳定的状态空间：能读取当前状态，能追溯历史变化，能按规则定时醒来，能把结果写回去，能让下一个人或下一个 Agent 接着工作。

文件夹正好是这样的空间。

一个需求可以是一个文件夹。PM、设计、研发、QA 都在同一个需求文件夹下工作；调研、截图、会议转写、PRD、设计说明、技术方案、测试用例都放在里面。README 是这个文件夹对外暴露的状态接口，Git 负责版本历史，Agent 可以定时读取、汇总、更新。

第一步并不复杂：只是把同一个需求的上下文放到同一个地方。

但这一步背后，其实藏着一个更大的变化：组织不再只靠人和人之间的消息传递来维持运转，而开始把局部工作状态沉淀为一个个可读取、可追溯、可运行的文件夹。

未来 AI 原生的组织，可能会越来越像一棵持续运行的文件树。

这些信号可以轻轻点到，不要写成“因为某工具支持所以我们要这么做”。更好的写法是：

> 工具侧已经开始出现运行时能力：Agent 可以定时醒来、读取项目上下文、执行任务、写回结果。组织的信息结构如果仍然停留在群聊和散落文档里，就很难承接这种能力。
