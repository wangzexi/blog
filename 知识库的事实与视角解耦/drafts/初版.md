# 知识库的事实与视角解耦

```text
Knowledge = Perspective(Facts)
```

知识不是事实本身，而是某种视角对事实的观察结果。

全文只讲两个关键词：事实和视角。

知识库真正应该长期保存的，不是某个固定视角下的总结、纪要、FAQ 或结论，而是两类东西：

```text
1. 尽量完整的事实层
2. 可以持续迭代的视角层
```

这就是事实与视角解耦。很多知识库效果不好，根源在于它们从一开始就把事实和视角绑定在了一起。

过去，人类阅读能力有限，检索能力有限，处理非结构化信息的能力也有限。一场会议结束后，需要有人写纪要；一个项目推进完，需要有人写总结；一次事故结束后，需要有人写复盘。原始聊天、会议录音、临时截图、日志、讨论过程太嘈杂，人不可能每次都从头读一遍。

于是，原始事实被压缩成文档。

文档不是事实。文档是某个视角下对事实的一次压缩。

## 一、传统知识库保存的是视角结果

以一场会议为例。

同一场会议，产品关心需求边界和优先级，研发关心接口、依赖和技术风险，测试关心验收条件和边界场景，运营关心上线节奏和用户影响。

如果会后只生成一份统一会议纪要，看起来所有人都有了共同结论，但这份纪要已经混入了记录者的视角。

记录者认为重要的东西被保留下来，记录者认为不重要的东西被丢掉。“重要”从来不是一个客观常量。

对产品不重要的细节，可能对研发很重要。对当前阶段不重要的信息，可能半年后复盘时非常重要。对普通成员不重要的争论过程，可能对新负责人理解历史决策非常重要。

传统知识库的典型路径是：

```text
原始事实
  -> 某个时刻的视角
  -> 固定文档 / 固定摘要 / 固定 FAQ
  -> 长期保存
```

它长期保存的不是事实，而是某个固定视角下的观察结果。

一旦角色、问题、组织阶段或判断标准变化，旧文档就会失效。新问题查不到，不是因为原始事实不存在，而是因为当初压缩时没人觉得它重要。

很多知识库越建越厚，却越来越不好用，就是因为它们缓存了太多结论，却丢掉了生成结论所需的原始事实。

## 二、事实和视角必须拆开

底层保存一份尽量完整的原始事实。上层根据不同角色、目标和问题，用不同视角重新观察这些事实。

```text
原始事实层
  会议、聊天、日志、代码、截图、文档、决策过程、用户反馈

视角规则层
  产品视角、研发视角、测试视角、运营视角、老板视角、新人视角、复盘视角

提问时生成
  当前问题需要什么，就从事实层里检索、筛选、组织和压缩什么
```

知识更像是在某个问题出现时，由当前视角从事实中临时生成的结果。

文档仍然重要，但位置变了。过去文档被当成知识本身；以后文档更像某个视角下的结果。

会议纪要是一种视角结果。项目总结是一种视角结果。README 是一种视角结果。周报、复盘、风险清单、FAQ，也都是视角结果。

真正稳定的，不应该是这些视角结果，而应该是它们背后的事实层。

## 三、现在可以保留更多事实

过去我们不得不提前压缩，是因为人直接处理不了高带宽的原始信息。

一个项目文件夹里如果塞满会议录音、群聊记录、截图、Excel、SQL、日志、设计稿、临时草稿，对人来说很快就是资料堆。东西是全的，但没人愿意看；材料是原始的，但噪声太大；每个角色只关心其中一部分，却要自己从噪声里筛。

过去更合理的做法，是把原始材料整理成文档，让人读文档，而不是读现实。

人不一定要直接读完整材料，而是可以带着问题问知识库：

```text
这个需求现在最大的风险是什么？
PRD 和最新群聊结论有没有冲突？
研发方案有没有漏掉设计约束？
QA 需要重点覆盖哪些边界？
这次事故从管理视角看真正的问题是什么？
新人接手这个模块，应该先理解哪几件事？
```

模型和 Agent 可以在提问时检索、比对、聚合、重组事实。

知识库的设计重心因此变化。

过去：

```text
提前整理好，方便人读。
```

现在：

```text
尽量保存事实，方便模型按不同视角重新观察。
```

存储、检索、按视角重新生成结果都变便宜了。真正昂贵的是原始事实：当时怎么讨论，为什么这么决策，有哪些反对意见，风险如何权衡，后来发生了什么。

这些事实一旦被压缩掉，很难再恢复。

## 四、不要用短生命周期视角污染长生命周期事实

事实层通常变化慢。

一场会议发生过就是发生过。一次事故的时间线、一次需求的讨论过程、一次代码变更的事实记录、一个用户反馈的原文，这些事实一旦产生，就成为历史。

视角变化快。

同一个角色在不同阶段会关心不同东西。新人研发关心功能能不能跑，高级研发关心可维护性和边界条件，架构师关心系统耦合和演化成本。公司早期关心增长，后期可能关心稳定、合规和效率。

```text
Facts 变化慢
Perspective 变化快
```

如果我们把一个快速变化的视角提前写进一个长期保存的知识文档里，本质上就是在用短生命周期的视角污染长生命周期的事实。

这会导致一个常见现象：知识库里明明有很多文档，但新的问题来了，还是查不到。不是没有资料，而是资料当初已经被某个固定视角压扁了。

这很像把 RAW 照片提前压成 JPEG。压缩后当然更小、更方便传播，但如果未来想重新调色、找暗部细节、恢复被丢掉的信息，就很难了。

如果只保存摘要，未来模型再强，也只能在旧摘要上推理。只有保留原始事实，未来才有机会从新视角重新解释。

## 五、知识库应该转向事实中心

传统默认：

```text
知识 = 文档
```

更合理的理解：

```text
知识 = 原始事实 + 当前视角
```

文档不是消失，而是从“唯一真相”降级为“某个视角下的结果”。

第一层是事实层。

```text
facts/
  chats/
  meetings/
  logs/
  screenshots/
  decisions/
  feedback/
  code-history/
```

它保存原始材料、过程、时间线和引用关系。目标不是写得漂亮，而是尽量不丢失真实事实。

第二层是视角层。

```text
perspectives/
  产品视角.md
  研发视角.md
  测试视角.md
  运营视角.md
  新人视角.md
  复盘视角.md
```

这些文档定义的不是“事实是什么”，而是“这个角色应该如何看事实”。

它描述：

- 关注什么
- 忽略什么
- 如何判断重要性
- 风险如何排序
- 如何压缩信息
- 输出应该用什么形式

表面上它还是 Markdown 文档，本质上是在描述一种视角。

它不是知识本身，而是从事实中生成知识的方式。

## 六、README 也是视角结果

`README.md` 很重要，但不应该被理解成文件夹的全部真相。

更合理的理解：

```text
文件夹 = 事实空间
README = 默认视角结果
即时问答 = 按需生成的视角结果
```

一个静态 README 可以提供稳定骨架，例如这个文件夹负责什么、当前状态是什么、依赖谁、风险是什么、谁在维护。

很多具体摘要不一定要人工提前写死。外部访问者可以根据自己的角色生成不同版本的 README。

上级看进度、风险和资源需求。同级看接口、依赖和变更影响。新人看背景、入口和关键文件。测试看边界、验收条件和风险点。Agent 看结构化状态、最近事件和下一步动作。

同一个文件夹，对不同访问者应该呈现不同视角结果。

一个团队内部有完整过程，对外只暴露必要状态。一个模块内部有大量细节，对外只暴露接口、能力和风险。一个人脑子里有复杂经验，对外表达时也会根据听众调整口径。

## 七、Agent 负责按视角读取事实

Agent 不是简单的总结工具。

它更像一个按视角读取事实的执行者。

```text
Agent = Perspective(Facts, Goal)
```

它读取事实层，理解当前目标，套用视角规则，生成当前需要的结果。

例如一个研发视角可以要求 Agent 关注：

```text
接口变更
技术风险
owner
blocker
可观测性
回滚方案
```

一个测试视角可以要求 Agent 关注：

```text
边界条件
异常路径
验收口径
回归范围
灰度风险
```

一个新人视角可以要求 Agent 关注：

```text
背景
核心概念
关键文件
历史决策
常见坑
学习路径
```

同一个事实库，经过不同视角，会生成不同知识。

Agent 时代知识库的价值，不是提前写出所有答案，而是保留一个可以被不同视角反复观察的事实空间。

## 八、知识库建设重点会改变

过去的问题：

```text
文档写得够不够完整？
分类清不清楚？
标签打得对不对？
FAQ 覆盖够不够多？
```

以后的问题：

```text
原始事实有没有被保留下来？
事实之间有没有基本时间线和引用关系？
不同角色的视角规则有没有沉淀？
Agent 能不能按角色生成不同视角结果？
旧结果能不能被新视角重新生成？
```

分类、标签、文档仍然有用，但不再是知识库的核心资产。

真正的核心资产是两类东西：

```text
1. 高质量事实层
2. 高质量视角层
```

事实层决定系统知道多少真实世界。

视角层决定系统如何看这些真实世界。

前者回答“发生了什么”，后者回答“应该怎么看”。

## 九、现实问题：原始事实不能只是垃圾堆

保留原始事实不等于把所有东西无脑塞进知识库。

企业里的原始材料经常噪声很大：聊天口语化、事实缺失、时间线断裂、信息互相矛盾、里面还有大量情绪和政治表达。

事实层需要最低限度的结构化。

至少保留：

- 时间
- 来源
- 参与人
- 所属项目或模块
- 事件关系
- 引用关系
- 决策和原因
- 后续结果

这里的结构化，不应该变成提前替事实下结论。

它更像是在给事实加坐标，让未来的检索和重组能够发生。

事实层要做的是保存和索引现实，不是提前解释现实。

## 十、最后

知识库的下一步，不是把文档写得更像文档。

而是把知识库从“结论仓库”变成“事实空间”。

过去，我们把原始事实经过某个视角压缩成文档，然后长期保存这个文档。

未来，更合理的方式是长期保存事实，把视角写成可迭代的规则，让 Agent 根据角色、目标和问题生成当前需要的结果。

> 知识库不应该保存某个固定视角下的世界，而应该保存可被重新观察的世界本身。
